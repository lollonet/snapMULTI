# ADR-003: Read-Only Container Filesystems

## Context

Running containers with writable root filesystems increases attack surface. A compromised process could modify binaries, install malware, or alter configuration.

## Decision

All containers run with `read_only: true`. Writable paths use size-limited tmpfs mounts.

## Consequences

### Positive
- Prevents runtime filesystem modifications
- Malware cannot persist across container restarts
- Forces explicit declaration of writable paths

### Negative
- Cannot use `docker cp` for hot-patching (must rebuild images)
- Some applications need workarounds (e.g., Tidal ALSA config via `ALSA_CONFIG_PATH`)
- tmpfs uses RAM, reducing available memory on constrained devices

### Mitigations
- tmpfs sizes are tuned per service (32M-64M)
- ALSA config workaround documented in CLAUDE.md
- Image rebuilds are automated via CI/CD tag push

## tmpfs Allocations

| Service | /tmp | /run | Other |
|---------|------|------|-------|
| snapserver | 64M | 64M | - |
| mpd | 64M | 32M | - |
| librespot | 32M | - | /config:1M |
| tidal-connect | 64M | - | /config:1M |
| others | 32M | varies | - |

# ADR-007: IPv4-Only LAN Appliance

## Context

snapMULTI is a single-LAN, single-purpose audio appliance. Every protocol it exposes or consumes runs over the local LAN: snapcast (TCP 1704/1705/1780), mDNS service discovery, MPD (6600), Tidal Connect + AirPlay + Spotify Connect mDNS publishes, the metadata-service WS/HTTP, snapweb, myMPD. There is no inbound or outbound traffic that requires IPv6 — Spotify/Tidal binaries make outbound TCP that resolves fine over IPv4.

Despite this, IPv6 was enabled by default on every install (kernel + interfaces + Avahi). That has caused recurring discovery failures:

- **Snapcast IPv6 link-local latch (PR #521)** — `snapclient`'s `libavahi-client` browse occasionally picks an IPv6 link-local SRV target that does not route on the local LAN. The device stays silent with a healthy server on the same LAN, visible to every other client.
- **`fb-display` IPv4 filter (PR #306 #9)** — `socket.inet_ntoa()` raises `OSError` on a 16-byte IPv6 entry. zeroconf returned v4+v6 in arbitrary order; fb-display crashed and skipped the v4 entry until the IPv4-only filter was added.
- **Avahi dual-publish race → `host-2.local` rename (issue #425)** — wlan0+eth0 dual publishes within a 6 s window before WiFi exclusivity disables the interface; the dual-stack version of the race is even harder to predict.

The pattern: every consumer of mDNS / Bonjour on snapMULTI needs hand-rolled protocol-purity defenses, and they get patched one at a time.

`tune_avahi_daemon` already sets `use-ipv6=no` to filter at the daemon level. That stops Avahi from publishing/browsing on v6, but does not stop other code paths (raw `getaddrinfo()`, libraries that bypass Avahi, NetworkManager auto-configuring link-local v6 on every interface). The daemon-side toggle is a partial fix.

## Decision

**Disable IPv6 at the kernel level on every snapMULTI install by default**, via `ipv6.disable=1` in `/boot/firmware/cmdline.txt`. Provide a documented opt-out (`DISABLE_IPV6=false` on the `prepare-sd.sh` / `.ps1` invocation) for the rare operator who has a hard requirement for IPv6 on the device.

The token is added by `prepare-sd.sh` / `prepare-sd.ps1` alongside the existing `systemd.mask=…` tokens (NetworkManager-wait-online, swap units), inside the same FAT32 boot-partition file. This places the disable at the earliest possible point in the boot sequence:

1. **PID 1 reads cmdline.txt before any unit starts.** The kernel sees `ipv6.disable=1` immediately and refuses to allocate the IPv6 stack. `socket(AF_INET6, …)` returns `EAFNOSUPPORT` everywhere from boot.
2. **No race with NetworkManager / Avahi / sysctl.** Userland tweaks (`net.ipv6.conf.all.disable_ipv6=1` via sysctl, NetworkManager `ipv6.method=disabled`) all run after PID 1 — there is a window where v6 was already published / discovered before the toggle landed. The kernel cmdline closes that window.
3. **Lives outside the overlayroot upper layer.** `/boot/firmware/` is FAT32, never wiped by overlayroot's tmpfs reset. A `systemctl mask` written to `/etc/systemd/system/` would be lost on every reboot once overlayroot is active; cmdline tokens survive ([[pattern_cmdline_mask_on_overlayroot]]).
4. **Reversible.** Removing the token from `cmdline.txt` and rebooting re-enables IPv6 fully — no migration script, no state to clean up.

### Hygiene that follows

- `Acquire::ForceIPv4 "true"` in `/etc/apt/apt.conf.d/99force-ipv4` — installed by `install-deps.sh` when `ipv6.disable=1` is on the kernel cmdline. Prevents 1-5 s per-fetch timeouts as apt tries AAAA against `debian.org` and Docker Hub mirrors before falling back.
- Docker daemon `ipv6: false` is already the default; no change needed.
- `tune_avahi_daemon`'s `use-ipv6=no` stays in place as belt-and-suspenders for the `DISABLE_IPV6=false` opt-out path.

### Workarounds retired by this decision

- **PR #521** — `discover-server-native.sh` + `ExecStartPre=+/usr/local/sbin/snapmulti-discover-snapserver` on Pi Zero 2W native installs. The script pinned `--host <IPv4>` into `/etc/default/snapclient` at every service start. With v6 disabled at kernel level snapclient's built-in `libavahi-client` browse can only see IPv4 records, so the `--host` pinning is redundant. The pinning also had a real cost: snapclient stopped auto-rediscovering the server when its IP changed, since `--host` overrides Avahi. Removing it restores native auto-recovery for free.

## Consequences

### Wins

- Eliminates an entire class of discovery bugs by removing the protocol the bugs depend on.
- Simpler reasoning during smoke checks / TROUBLESHOOTING: "is this device v4 or v6?" stops being a question.
- Less Avahi traffic (daemon handles half the records).
- Lower mental overhead in code review — protocol-purity defenses become unnecessary.

### Trade-offs (with mitigations)

| Trade-off | Mitigation |
|-----------|------------|
| go-librespot / Tidal Connect outbound goes v4 only. | Both providers run dual-stack; v4 fallback is universal and used by the majority of clients today. No functional regression. |
| `apt` may try AAAA and time out per fetch. | `99force-ipv4` apt config installed automatically when the kernel disable is active. |
| Code hardcoding `::1` instead of `127.0.0.1` would break. | Audited: `metadata-service.py` and `fb_display.py` use `gethostbyname` / `127.0.0.1`, no `::1` literals. |
| Operator on a v6-only ISP cannot reach internet streaming services. | `DISABLE_IPV6=false` opt-out, documented in ADVANCED.md + TROUBLESHOOTING.md. |
| "Disabling IPv6 is bad practice in 2026" reflex objection. | snapMULTI is a LAN-only audio appliance, not a general-purpose host. The doctrine that applies to laptops and servers does not apply to embedded appliances on a closed protocol surface. |

### Non-trade-offs (not a problem)

- Docker bridge networking is IPv4 by default. Containers using host networking inherit the host (IPv4 only after this change). `fb-display` uses bridge — IPv4 only.
- NTP sync: `pool.ntp.org` dual-stack; v4 fallback is instant.
- mDNS still works: `_snapcast._tcp`, `_airplay._tcp`, `_spotify-connect._tcp` are all advertised on IPv4 by both publisher and consumer.

## How to apply

| Surface | Change |
|---------|--------|
| `scripts/prepare-sd.sh` | `cmdline_add_token "ipv6.disable=1"` after the BOOT_MASK_UNITS loop, gated on `DISABLE_IPV6=true` (default). |
| `scripts/prepare-sd.ps1` | Mirror of the above, gated on `$env:DISABLE_IPV6` (default `'true'`). |
| `scripts/common/install-deps.sh` | Install `99force-ipv4` apt config when `ipv6.disable=1` is on `/proc/cmdline`. |
| `client/common/scripts/setup-zero2w.sh` | Remove ExecStartPre + `discover-server-native.sh` install (retired workaround). |
| `client/common/scripts/discover-server-native.sh` | Deleted. |
| `docs/ADVANCED.md` + `.it.md` | Document `DISABLE_IPV6=false` opt-out + reasoning. |
| `docs/TROUBLESHOOTING.md` + `.it.md` | "Why is IPv6 off on my snapMULTI?" + how to re-enable. |

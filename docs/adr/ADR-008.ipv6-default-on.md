# ADR-008: IPv6 Enabled by Default — Rely on Software Defenses, Not Kernel Kill-Switch

## Context

[ADR-007](ADR-007.ipv4-only-lan-appliance.md) disabled IPv6 at the kernel level (`ipv6.disable=1` in `/boot/firmware/cmdline.txt`) on every snapMULTI install to close three observed races:

1. **Snapcast IPv6 link-local latch (PR #521)** — `snapclient` libavahi-client picked an IPv6 link-local SRV target that did not route.
2. **fb-display crash on IPv6 entry (PR #306)** — `socket.inet_ntoa()` raised on 16-byte IPv6 addresses returned by zeroconf.
3. **Avahi dual-publish race (issue #425)** — wlan0+eth0 dual publishes within a 6-sec window before WiFi exclusivity kicks in.

The reasoning was "belt and suspenders": kernel kill-switch + software defenses for redundancy.

## What changed

Live test on snapvideo (Pi 4 8 GB, both-mode) on 2026-05-29:

1. Removed `ipv6.disable=1` from cmdline.txt via live edit, rebooted.
2. IPv6 stack came up; Avahi config (`use-ipv6=no` in `/etc/avahi/avahi-daemon.conf`) correctly published IPv4-only SRV records for snapcast / AirPlay / Spotify / Tidal Connect.
3. snapclient peers (`snapdigi`, `pizero` — both still running with `ipv6.disable=1`) reconnected to snapvideo without issues — no latch on link-local.
4. **Tidal Connect WebSocket server now starts cleanly.** Previously it failed with `Address family not supported by protocol` because the proprietary `speaker_controller_application` binary requires `socket(AF_INET6)` for its `[::]` WebSocket listen even though audio routes over IPv4 only.
5. fb-display rendered cover art without zeroconf crash (PR #306 IPv4 filter holds).
6. No `host-2.local` rename on first boot (the wlan0+eth0 dual-publish race did not fire on this WiFi-only host; on a dual-iface host, `boot-tune.sh`'s WiFi-disable-when-eth0-up still closes that window).

The kernel kill-switch was **redundant** with the software defenses, and it broke a working upstream feature.

## Decision

**Ship with IPv6 ENABLED at the kernel level by default.** The three software defenses carry the load:

| Defense | Closes |
|---|---|
| Avahi `use-ipv6=no` in `/etc/avahi/avahi-daemon.conf` (server + client) | Snapcast / AirPlay / Spotify / Tidal mDNS SRV publish stays IPv4-only |
| `discover-server.sh` (PR #521 logic, installed as `/usr/local/bin/snapclient-discover`, `ExecStartPre=` on `snapclient.service`) | Residual link-local latch on the containerized-client side. Filters `avahi-browse -rpt _snapcast._tcp` output on `$3=="IPv4"` before picking a host. Pi Zero 2W native snapclient skips this script (commented as a deliberate trade-off in the file) and relies on the Avahi `use-ipv6=no` defense at the server. |
| PR #306 — `socket.inet_ntoa` IPv4-only filter in fb-display zeroconf result loop | IPv6 entry crash defensive |
| `boot-tune.sh` — disable WiFi when eth0 carrier is UP | Avahi dual-publish race on dual-iface hosts (issue #425) |

Provide a documented opt-IN (`DISABLE_IPV6=true` on the `prepare-sd.sh` / `.ps1` invocation — note: inverted polarity from ADR-007's `DISABLE_IPV6=false`) for the rare operator who has a hard reason to disable.

## Why this is safe

The original ADR-007 worry was "userland tweaks have race windows; kernel cmdline is the only point that pre-empts every userland subsystem". That's true — but the software defenses don't operate during the kernel-to-userland transition. They operate AFTER the relevant userland subsystem starts:

- Avahi's `use-ipv6=no` filter is checked at daemon start, before any record publish.
- snapclient's IPv4 SRV pin is enforced via ExecStartPre, before the binary calls libavahi-client.
- fb-display's zeroconf filter is in the Python event loop, applied to every result.

No race window exists where any of these defenses can be bypassed by an upstream stack timing. The kernel disable was extra insurance against a failure mode that didn't actually occur in this codebase.

## Consequences

**Pro**:

- Tidal Connect works out-of-box on every install. No need for the live cmdline edit or per-device opt-out.
- Future upstream binaries that default-bind `[::]` (myMPD was the first casualty, fixed via `MYMPD_HTTP_HOST=0.0.0.0`; Tidal was the second) don't require per-binary workaround.
- Aligns with Raspberry Pi OS default (IPv6 on).
- iOS / macOS / Android casting tools that prefer dual-stack don't hit fallback paths.

**Con**:

- One more variable to keep an eye on if Snapcast's mDNS browsing ever regresses. The software defenses must continue to hold.
- `apt`'s `Acquire::ForceIPv4 "true"` config (auto-installed when `ipv6.disable=1` was on cmdline) is no longer needed by default. Removed correspondingly from the install-deps install path.
- Operators on legacy networks where IPv6 router advertisements misbehave may want the opt-in disable. Documented in ADVANCED.md.

|---|---|

| `scripts/prepare-sd.ps1` | Mirror of the above. |

No changes to:

- Avahi config (`use-ipv6=no` stays — it was already the load-bearing defense for IPv6 link-local SRV).
- `boot-tune.sh` (wlan0+eth0 single-publish race stays closed).
- snapclient / fb-display IPv4 pins / filters (defensive, already in place).

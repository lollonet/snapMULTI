# ADR-004: Centralized Metadata Service

## Context

Each audio source has its own metadata mechanism (MPD protocol, Spotify WebSocket, AirPlay pipe, Tidal TUI scraping). Clients need cover art and track info for their current stream. Previously, each client ran its own metadata-fetching containers.

## Decision

A single server-side metadata service polls all sources and pushes metadata to clients via WebSocket. Cover art is fetched once and served via HTTP.

## Consequences

### Positive
- Clients need zero metadata containers (2 fewer per client)
- Cover art fetched once, shared across all clients
- Consistent metadata format regardless of source
- Single point for rate-limited API calls (MusicBrainz, iTunes)

### Negative
- Single point of failure for metadata (audio continues without it)
- Additional server-side container (memory/CPU overhead)
- WebSocket connection management complexity

### Architecture

```
Sources → controlscripts → Snapserver JSON-RPC
                                    ↓
                          metadata-service.py
                           ↙          ↘
                     WS:8082        HTTP:8083
                   (push to        (artwork
                    clients)        files)
```

### Cover Art Chain
1. MPD embedded art (local files)
2. iTunes Search API (free, no key)
3. MusicBrainz Cover Art Archive (rate-limited 1 req/s)
4. Radio-Browser API (for radio streams)

### Mitigations
- Graceful degradation: audio plays without metadata
- MusicBrainz smart rate limiter (sleeps only remaining time)
- Reconnection logic for snapserver connection loss
- Consecutive error counter with auto-restart

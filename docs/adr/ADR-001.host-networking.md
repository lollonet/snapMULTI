# ADR-001: Host Networking for Audio-Path Containers

## Context

Snapcast clients discover the server via mDNS (multicast DNS). Audio source services (Spotify, AirPlay, Tidal) also use mDNS to advertise themselves to controlling apps. Docker bridge networking isolates containers from the host network, breaking multicast.

## Decision

All containers in the audio path use `network_mode: host`:
- **Server stack (7/7)**: snapserver, mpd, mympd, shairport-sync, librespot, tidal-connect, metadata
- **Client stack (2/3)**: snapclient, fb-display

`audio-visualizer` is the documented exception — it serves a WebSocket spectrum feed on port 8081 to fb-display on the same host, has no multicast/mDNS need, and stays on Docker's default bridge networking. Keeping it isolated reduces host-port surface area for no functional cost.

## Consequences

### Positive
- mDNS/Avahi works natively — zero-configuration discovery
- No NAT overhead on audio streams
- Simpler debugging (services visible on host interfaces)

### Negative
- Port conflicts possible with host services
- No network isolation between containers
- Container ports visible on all host interfaces (LAN exposure)

### Mitigations
- Fixed port allocation per service (table in [`docs/USAGE.md`](../USAGE.md#architecture))
- deploy.sh checks for port conflicts
- Designed for dedicated appliance use, not shared servers

## Alternatives Considered

- **Bridge + macvlan**: Would give containers their own IPs, but requires network-specific configuration and breaks the zero-touch promise
- **Bridge + manual mDNS proxy**: Complex, fragile, and adds latency

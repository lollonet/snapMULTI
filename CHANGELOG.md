### Changed
- **`scripts/firstboot.sh:run_checkpointed_phase` helper** (v0.8 hardening track, PR5). Extracts the repeated `if checkpoint_reached "X"; then log_info "skip"; else <body>; checkpoint_done "X"; fi` pattern into a single function. Two of the six checkpoints migrated (`deps`, `docker`) â€” clean linear bodies that map cleanly to `_phase_<name>()` function dispatch with bash's shared-scope semantics (the `DOCKER_REPO_PRECONFIGURED` mutation in `_phase_deps` still propagates to the docker block as before). Helper preserves checkpoint marker names byte-identically (same `.done-<name>` files, same log messages, same `set -euo pipefail` propagation, no subshell â€” variable mutations stay in caller scope). New `tests/test_firstboot_checkpoint_helper.sh` (16 checks): static migration assertions (`deps` + `docker` use the helper, legacy `if checkpoint_reached` form removed for those two, four others stay inline, checkpoint name inventory frozen), functional unit tests via standalone extract (success â†’ marker + rc=0, failure â†’ no marker + propagated rc, skip â†’ body not invoked + log emitted, mutation â†’ propagates without subshell). ## [0.7.9.7] â€” 2026-06-01

> Script-only patch (image_set stays 0.7.7). Closes the production reboot loop on Ethernet+WiFi-off hosts (#566 â€” observed 4 reboots in 40 min on snapvideo), reconciles Avahi `allow-interfaces` at every boot to handle topology changes (#565), fixes the boot-tune eth0 DHCP race that left both interfaces active (#564). ### Fixed
- **WiFi watchdog reboot loop on Ethernet hosts with admin-disabled WiFi** â€” `snapmulti-wifi-watchdog.service` has `After=NetworkManager.service network-online.target`, but snapMULTI's `cmdline.txt` masks `NetworkManager-wait-online.service` (intentional, to avoid 90 s boot delay). Result: `network-online.target` fires immediately at boot, before NM has assigned an IPv4 to eth0. The one-shot eth0 IPv4 probe returned false â†’ script proceeded to the main loop â†’ `resolve_target()` picked the eth0 gateway as fallback â†’ `ping -I wlan0` failed every poll (wlan0 admin-down via `nmcli radio wifi off`) â†’ 10 Ă— 60 s = `systemctl reboot` â†’ reboot loop every 10 min. Observed live on snapvideo 2026-05-31: 4 reboots in 40 min. Fix: wrap the eth0 IPv4 probe in a 30 s retry loop (handles DHCP race) and add a defense-in-depth check for `nmcli radio wifi == disabled` (graceful exit when there's no WiFi to watchdog). snapdigi + pizero are WiFi-only and were unaffected. 5 new static invariants pin the retry loop, its bound, the `ip link show eth0` gate (no 30 s delay on WiFi-only hosts), and the nmcli radio-disabled check.
- **Avahi `allow-interfaces` left stale across network topology changes â€” mDNS muto when the chosen interface goes DOWN** â€” `tune_avahi_daemon` (from `system-tune.sh`) decides which interface Avahi can publish on (wired-carrier priority, with fallback to default-route enumeration). That decision used to happen ONCE at firstboot, then never again. Observed live on snapvideo 2026-05-31: firstboot ran without eth0 cable â†’ `allow-interfaces=wlan0`; user later attached eth0 + manually disabled WiFi â†’ wlan0 DOWN â†’ Avahi muto on both interfaces (the only allowed one was DOWN). `snapvideo.local` resolution timed out from the host itself; the `/status` page falsely reported "mDNS round-trip OK" because the check only looks at avahi-daemon liveness, not interface-binding correctness. Fix: `boot-tune.sh` now sources `system-tune.sh` and re-runs `tune_avahi_daemon` after the WiFi disable/enable decision is settled. Idempotent (no restart when nothing changed). 23 new static invariants pin the source path candidates, the `declare -F` guard before calling, the non-fatal fallback, the position after the WiFi block (so carrier state is settled), and that `tune_avahi_daemon` still self-restarts avahi-daemon when the config actually changes.
- **boot-tune.sh race against eth0 DHCP â€” both eth0 and wlan0 left active, dual-mDNS risk** â€” `snapmulti-boot-tune.service` runs `After=docker.service` only. On hosts where NetworkManager activates eth0 DHCP in the same boot window (observed on snapvideo 2026-05-31: eth0 `activated` at 01:16:45 â†” boot-tune started at 01:16:45 same second), the one-shot `ip -4 addr show eth0` probe saw eth0 without an IP yet â†’ kept WiFi ON â†’ both interfaces came up with IPs â†’ dual-mDNS risk + `192.168.63.4 conflict with eth0 MAC DC:A6:32:B4:C6:16` in NM logs. Fix: gate the probe on `/sys/class/net/eth0/carrier == 1` (cable plugged in), then loop for up to 15 s waiting for DHCP to actually assign an IP. WiFi-only hosts (no eth0) short-circuit cleanly. 10 new static invariants pin the carrier gate, the bounded loop (â‰Ą 10 s), the `break` on first IP, and the unchanged `wifi off` / `wifi on` branches.

### Fixed
- **backup scripts no longer degrade `/status` for 24 h on transient remount failure** â€” `backup-mpd.sh`, `backup-snapmulti-state.sh`, and `save-diagnostics.sh` all called `mount -o remount,rw /boot/firmware 2>/dev/null || true` then proceeded blindly. When the remount silently failed (observed on snapvideo 2026-05-29 first boot â€” service exited with `mkdir: cannot create directory: Read-only file system` at 15:02), the unit landed in `failed` state and the `/status` page surfaced `systemd state degraded â€” 1 unexpected failed unit(s): snapmulti-backup.service` until the next manual `reset-failed` or daily timer success. All three now validate the remount via `findmnt` and exit 0 with a journal entry when the partition is still ro â€” best-effort backup, transient failure does not degrade the overall system health signal. Root cause of the original remount failure remains under investigation; this fix decouples backup transient errors from `/status` fail signaling.
- **metadata-service `/version`: semver tuple comparison for `update_available` instead of `!=`** â€” observed on snapvideo with `SNAPMULTI_RELEASE=v0.7.9.5` in `.env` while GitHub `releases/latest` returned `v0.7.9.3` (v0.7.9.4 + v0.7.9.5 were tag-without-publish). The old `latest != current` flagged `update_available=true`, surfacing the misleading "Update available: 0.7.9.5 -> 0.7.9.3 (reflash to apply)" on the `/status` page. `_parse_version_tuple()` now converts both sides to int tuples and the check becomes `latest_tuple > current_tuple`. Pre-release tags / dev clones (non-numeric parts) fall back to the old `!=` comparison so the endpoint never crashes. 11 new regression tests pin the matrix (current ahead, current behind, equal, unparseable, major/minor jumps).
- **install-deps: `apt-get upgrade` â†’ `apt-get dist-upgrade` so kernel + rpi-eeprom are not kept back** â€” plain `upgrade` refuses to install new packages or remove old ones, so meta-packages like `linux-image-rpi-2712` / `linux-image-rpi-v8` / `rpi-eeprom` are silently held back when the new version requires installing a new sub-package (the new kernel image deb). Observed on snapvideo 2026-05-29: 6.12.75 â†’ 6.18.29 kernel + 28.15 â†’ 28.24 eeprom all stuck after a clean reflash. Fix applies to both `install-deps.sh` and the final upgrade pass in `firstboot.sh`. `apt-get dist-upgrade -y` is the standard for "actually apply pending kernel + eeprom updates".
- **install-deps: drop `dstat` from monitoring tools (eliminates pcp entirely)** â€” `dstat` Depends on `python3-pcp`, which transitively pulls Performance Co-Pilot (~50 MB, multi-daemon). On Debian trixie neither `sysstat` nor `iotop-c` Depends on pcp (verified: `apt-cache depends iotop-c` shows only libc6, libncursesw6, libtinfo6), so dropping `dstat` means new installs no longer pull pcp at all â€” no transient install, no `systemd-sysv-generator: SysV service '/etc/init.d/pcp'` journal noise. `sar` from sysstat covers dstat's realtime CPU/mem/disk use case.
- **install-deps: explicit `apt-get purge pcp pcp-conf 'pcp-pmda-*'` removed (dead code after dropping dstat)** â€” with pcp no longer pulled by any package we install, the trailing purge has nothing to clean. The existing `apt-get autoremove --purge -y` (run earlier in the same function) acts as the safety net for any orphan pcp left over from a base Pi OS image or a stale state. -7 LOC, comment block rewritten to document the new invariant.

> Script-only patch (image_set stays 0.7.7). Resolves the v0.7.9 persistence regression. Field-validated on snapvideo: overlay active + `server.json` + `server.json.prev` + 4 groups all survived reboot under `overlayroot="tmpfs"`.

See [docs/ADVANCED.md â€” MPD_START_PERIOD](docs/ADVANCED.md#mpd_start_period) and [docs/TROUBLESHOOTING.md â€” Install marked failed but containers run](docs/TROUBLESHOOTING.md#install-marked-failed-but-containers-run).

### Fixed
- **v0.7.9 persistence regression cascade (#527)** â€” multiple compounding bugs from the v0.7.9 persistence work, all corrected here. The original `snapmulti-data-persistence.service` bind-mounted `/opt/snapmulti/data` over `/media/root-rw/snapmulti-persist/`, but `/media/root-rw/` is itself tmpfs on `overlayroot="tmpfs"` â€” the bind worked at runtime but the persistent location was wiped at reboot, and the smoke check `[OK] Persistent bind active` actively masked the failure. Replaced with the proven `/boot/firmware/`-backup pattern (FAT32, the only userland-writable path that survives reboot on tmpfs overlayroot):
  - **Backup trigger** is split: `snapmulti-state-backup.path` watches myMPD workdir (event-driven, ~1 s latency for playlist/theme/script edits); `snapmulti-state-backup.timer` runs every 5 min for `server.json` (snapserver rewrites that file every ~3 s to refresh client `lastSeen` heartbeats â€” wiring it to `.path` would cause `~354` backup-service invocations per 5 min and churn the FAT32 boot partition). `backup-snapmulti-state.sh` short-circuits on canonical equality (sort keys + strip `lastSeen` + sha256 compare) so a no-op timer tick costs only one JSON parse â€” zero FAT32 writes when only heartbeats changed.
  - **Restore** runs as `ExecStartPre` on `snapmulti-server.service` BEFORE compose up, so snapserver finds restored state instead of writing a fresh empty one. The script PUID/PGID is read from `/opt/snapmulti/.env` for non-default user installs.
  - **Scope** matches the original bind-mount: snapserver `server.json` + WHOLE `myMPD/workdir/` (state + config + smartpls + scripts + pics). Narrowing to `state/` alone would silently lose user-customised smart playlists / theme / scripts on reboot.
  - **Seed on install** runs `snapmulti-state-backup.service` once when arming the watcher â€” protects upgrade scenarios where state pre-exists the watcher and would otherwise leave the boot-partition backup stale until the next user change.
  - **Boot partition mount state preserved** â€” `backup-snapmulti-state.sh` + `backup-mpd.sh` detect whether `/boot/firmware` was previously ro and restore THAT state on exit. The original v0.7.9 fix unconditionally remounted ro at exit, which during firstboot (both-mode) cascaded to break `raspi-config do_overlayfs` and prevented overlay activation entirely â€” discovered by user pattern-recognition ("readonly in server mode has always worked").
  - Removed `snapmulti-data-persistence.service`, `snapmulti-data-setup.sh`, and the misleading `check_persistence.sh` smoke module; replaced with `check_state_backup.sh` (verifies backup freshness on the actual persistent path) + `check_timers.sh` now also asserts the path + timer units are enabled and active.
  - **`backup-from-sd.sh` catches up to widened backup scope (#530)** â€” host-side reflash extractor now handles the current `mympd/workdir/` layout (atomic stage+swap to avoid merging stale files from the previous SD with the restored backup) and restores `data/server.json`. Legacy `mympd/state/`-only backups still work with a warning that user customisations aren't in the backup.
  - **Backup integrity + `.prev` rotation** â€” `backup-snapmulti-state.sh` validates each `server.json` write (size â‰Ą 64 B + JSON parse via `jq` when available) before publishing, and rotates the previous good copy to `server.json.prev` so a single transient corruption (truncated write, FAT32 fragmentation, power loss mid-publish) cannot brick the device. `restore-snapmulti-state.sh` validates `server.json`; falls back to `.prev` with a WARN log if current is corrupt; fails loud only when BOTH are unparseable. Plus a post-`verify_services` seed in `deploy.sh` so the boot-partition backup captures real state on first install (the in-place seed at install time was a no-op on fresh installs because snapserver hadn't written yet). 2 new bash tests (`test_backup_integrity_validation.sh`, `test_restore_prev_fallback.sh`) lock the lessons in CI.
- **`verify_services` healthcheck stability grace (120 s)** â€” Docker does not necessarily mark a container healthy the instant the inner service binds its port; the next health probe runs up to 30 s later, and `verify_compose_stack` also needs a second sample to prove `RestartCount` stability. Large NFS libraries were observed to bind just after the 300 s start_period and then fail `verify_services` seconds before the next successful probe. New explicit `health_grace_seconds=120` on top of `max(MPD_START_PERIOD, 180s)` covers that window. The install log now exposes the full budget (`up to ${verify_budget}s, ${wait_seconds}s interval; MPD start_period=${start_period}s + 120s grace`). Does not eliminate the need to bump `MPD_START_PERIOD` for very large libraries â€” see Known limitation above.
- **Mount smoke accepts active `.mount` as valid (`check_mounts.sh`)** â€” some systemd versions mark the `.automount` companion inactive once the actual `.mount` has fired (e.g. after MPD scanned the library). Previously this scored as a fail. Now passes when either the automount is watching OR the mount has already fired; only fails when neither holds.

## [Unreleased]

### Fixed
- **`client/common/scripts/setup.sh` ‚Äî `snapclient.service` recreate guard now uses runtime probe instead of `/var/lib/` sentinel** (fix on top of PR #393). The previous sentinel `/var/lib/snapclient/.mem-recreate-done` lived under the overlayroot tmpfs upper layer (`tmpfs:recurse=0` overlays everything except `/boot/firmware`), so it disappeared on every reboot ‚Äî defeating the "exactly once" intent and recreating the container on every restart. The new `ExecStartPre` probes the live container with `docker inspect snapclient --format '{{.HostConfig.Memory}}'`: returns `"0"` when limit is missing (recreate), the byte count once recreated (skip), and empty when the container does not yet exist on a fresh reflash (skip ‚Äî ExecStart will create it correctly). No persistent state required.
- **`scripts/smoke/check_audio_modules.sh` skips cleanly on Bash <4** ‚Äî the module uses `declare -A` for the HAT ‚Üí kernel-module table. Pi OS Bookworm ships Bash 5.2 (fine), but macOS `/bin/bash` is 3.2 and chokes with `unbound variable` errors that drown the contributor's test run. A `BASH_VERSINFO[0] < 4` guard at the top emits a single `SKIP:` line and returns, so the rest of `tests/run-all-tests.sh` reports clean on macOS. The module remains fully functional on the Pi where it actually runs.
- **`tests/test_install_pipeline_wave2.sh` asserts the correct RequiresMountsFor invariant** ‚Äî the test comment and primary assertion claimed `deploy.sh adds network MUSIC_PATH to RequiresMountsFor`, which is the OPPOSITE of the current design (`deploy.sh:1010` explicitly does NOT add network paths to `RequiresMountsFor`, leaving them to systemd `.automount` lazy mount). The test passed only because the pattern matcher was too broad. Updated to verify three real invariants: (a) `music_mount_clause` is initialised empty, (b) the `nfs|smb|network` branch explicitly says `NOT adding ... RequiresMountsFor`, (c) the `is_network_mount` runtime probe remains as fallback when MUSIC_SOURCE is unset. The test now protects the regression.
- **`README.md` + `README.it.md` "No cloud, no subscription" claim softened** ‚Äî Spotify Connect requires a Premium subscription, Tidal Connect requires a Tidal account, AirPlay requires an Apple device. The previous absolute claim invited easy fact-check pushback on Reddit / HN. Reworded to "Core runs locally, no snapMULTI cloud or telemetry; the streaming services (Spotify Premium, Tidal, AirPlay) keep their own account / device requirements" ‚Äî same message about snapMULTI being local-first, honest about the upstream services. Italian mirror synced.
- **`client/common/scripts/setup.sh` ‚Äî HAT detect + config.txt write visibility + fail-loud on missing dtoverlay** ‚Äî observed live on pi3hat (Pi 3B+, InnoMaker DAC, EEPROM-less): install completed without error, but snapclient went into restart-loop with `Can't open hw:CARD=sndrpihifiberry,DEV=0, error: No such device, code: -19`. `config.txt` was left factory-default ‚Äî no `dtoverlay=hifiberry-*` written ‚Äî and ALSA only saw `Headphones` + `vc4hdmi`. The bundled `install.log` was silent on the entire HAT path because every diagnostic `echo` in `setup.sh` (`Auto-detected HAT:`, `Backup created:`, `Removing previous snapclient configuration...`, `Disabled otg_mode=1`, `Loaded overlay ‚Ä¶ at runtime`) was a plain echo with no `[INFO]/[OK]/==>` prefix, so the firstboot.sh case filter dropped them all silently. Root cause was thus invisible from the bundle. Fix: (a) add `[INFO]` prefix to every load-bearing diagnostic echo so the firstboot filter captures them; (b) add an explicit `[INFO] HAT decision: config=‚Ä¶  overlay=‚Ä¶  card=‚Ä¶  source=‚Ä¶` line right after the HAT config file is sourced ‚Äî single line that captures everything a future debugger needs; (c) add post-write validation after the dtoverlay block lands in `config.txt` ‚Äî if `grep -qF "dtoverlay=$HAT_OVERLAY"` fails on `$BOOT_CONFIG`, `exit 1` with a clear `ERROR:` banner instead of letting the install continue toward a guaranteed runtime failure. Skipped for `HAT_OVERLAY=""` (usb-audio, internal-audio) where no overlay write is expected.
- **`client/common/scripts/setup.sh` ‚Äî `snapclient.service` recreates the container on the first start post-install** so the `mem_limit` / `cpu_limit` declarations in `docker-compose.yml` take effect. During first boot `firstboot.sh:1068` starts the snapclient container BEFORE the final reboot, but `cmdline_ensure_memory_cgroup` (PR #389) takes effect only AFTER reboot ‚Äî so the container is created without cgroup memory v2 and `HostConfig.Memory=0`. The subsequent `snapclient.service` ExecStart (`docker compose up -d` plain) is idempotent and keeps the limit-less container. Smoke caught this on snapvideo first successful reflash: `Container(s) with HostConfig.Memory=0 (limit drift ‚Äî deploy.sh must --force-recreate): snapclient`. Fix: a sentinel-gated `ExecStartPre` runs `docker compose up -d --force-recreate` exactly once after install and touches `/var/lib/snapclient/.mem-recreate-done`. Subsequent boots find the sentinel and skip ‚Äî no container churn on regular restarts. Server-side containers don't have the issue because `deploy.sh:910` already does `--force-recreate` during install.
- **`firstboot.sh` deploy / setup case filters now surface Docker Compose state milestones** ‚Äî the install log appeared frozen for 2‚Äì4 minutes during the first-boot image pull and `docker compose up`, because the case statement at `:797-810` (and the mirror at `:996-1010`) accepted only `[INFO]`/`[OK]`/`==>` lines from the subprocess and silently dropped everything else. Docker Compose state-transition lines (`Container snapserver Started`, `Container metadata Healthy`, `Network snapmulti_default Created`) carry no such prefix and went straight to the void, while the per-layer `Pulling`/`Pull complete` spam (intentionally dropped) gave the impression that nothing was happening. Both filters now match `Container <name> Started/Created/Healthy/Running/Stopped/Removed` and `Network/Volume <name> Created`, so the user sees the install advance through container milestones in the TUI even during long pulls. Per-layer pull progress remains suppressed (it's noise, not signal). Verified live: matches the symptoms reported on snapvideo first-boot ("log non avanza, install boh") and a third snapvideo reflash.
- **Community-launch blocker quintet** ‚Äî 5 fixes gathered from cross-audit (7-agent council + 2 independent external reviewers):
  - **`client/common/scripts/setup.sh:1231` ‚Äî `Requires=avahi-daemon.service` demoted to `Wants=`** to match the server pattern (`scripts/deploy.sh:1026-1028`). With `Requires=`, an avahi-daemon flap (common on flaky WiFi or Pi Zero 2W BCM43430 quirks) propagated a failed state to `snapclient.service` and blocked startup, even in both-mode with a local server. The `ExecStartPre` readiness loop (bounded 30 s sleep) was already non-fatal ‚Äî `Requires=` made the bounded wait useless. `PartOf=avahi-daemon.service` is retained (propagates restarts for the upstream libavahi-client bug fix).
  - **`scripts/prepare-sd.ps1` did not copy `diagnostic.sh`** on either the server or client side. The bash version (PR #380) did; the Windows mirror had been forgotten. Consequence: users preparing the SD from Windows lost the `install-failed-*.tar.gz` bundle on the FAT32 boot partition when the install failed ‚Äî that bundle is the main recovery path without SSH. Added to both `Copy-ServerFiles`/`Copy-ClientFiles` and to the verify list.
  - **`scripts/firstboot.sh` ‚Äî `scrub_credentials` moved from post-deploy-verify to post-mount-music**: the previous position (line 831, after `verify_compose_stack`) meant any `deploy`/`verify` failure left `SMB_PASS`/`SMB_USER`/`NFS_SERVER` plaintext on `/boot/firmware/install.conf` (FAT32, readable from any PC after the SD is pulled). The scrub now runs immediately after `setup_music_source()` ‚Äî mount-music has already persisted the credentials to the systemd `.mount` units on ext4 (root-only), so `install.conf` on FAT32 no longer needs them. A new `music` checkpoint guarantees that a post-crash retry does not re-enter `setup_music_source()` with the empty creds (the issue documented by the prior comment). Scrub is positioned OUTSIDE the if/else so a crash between `checkpoint_done "music"` and the scrub itself does not leave creds plaintext forever ‚Äî scrub is idempotent so unconditional execution is safe.
  - **`scripts/common/sanitize.sh` ‚Äî `sanitize_nfs_export` / `sanitize_smb_share` now fail loud on spaces** instead of silently stripping. Common NAS units (Synology, QNAP) default to shares like `/volume1/Music Share`; `tr -cd 'A-Za-z0-9/._-'` produced `/volume1/MusicShare` and the mount then failed with a confusing ENOENT ‚Äî the user thought it was a network / permissions issue when in fact the script had corrupted the path. The functions now emit 3 ERROR lines to stderr explaining the problem ("Rename the share on the NAS e.g. 'Music_Share' and retry") and return rc=1 / empty stdout, so the caller (firstboot, prepare-sd) fails immediately at the source.
  - **`docker-compose.yml` ‚Äî `shairport-sync` + `librespot` now bind-mount `/etc/hostname:/etc/hostname:ro`**: `network_mode: host` shares only the NET namespace, not UTS, so `$(hostname)` inside the container returned the 12-character container ID (e.g. `a1b2c3d4e5f6`). The fallback device names for AirPlay (`airplay-entrypoint.sh:17 ‚Üí "$(hostname) AirPlay"`) and Spotify (`docker-compose.yml:125 ‚Üí "$(hostname) Spotify"`) produced names like "a1b2c3d4e5f6 AirPlay" in the client app instead of "snapvideo AirPlay". `tidal-connect` already had the correct bind-mount ‚Äî the pattern was copied.
- **`scripts/deploy.sh` + `scripts/status.sh` legacy color refs ‚Üí `unbound variable` install abort** ‚Äî Bundle B1 consolidated `logging.sh` into `unified-log.sh` and renamed the exported color variables from `GREEN` / `RED` / `BLUE` / `BOLD` / `NC` to the `_LOG_*` prefixed form, but four call sites in `deploy.sh` (lines 112, 129, 947, 965) and the TTY-guard in `status.sh:19` kept referencing the old names. Under `set -u` (firstboot install chain), the first hit aborts the whole script with `scripts/deploy.sh: line 947: GREEN: unbound variable`. Observed live on snapvideo first reflash with v0.7.4 ‚Äî install reached the post-deploy `show_status` banner, hit the unbound variable, exited rc=1, fired the `cleanup_on_failure` trap, and produced an install-failed bundle. PR #389's install.log capture is what made root-causing possible in 30 seconds (line 464 of the bundled install.log: the unbound-variable trace). Same family as PR #387 which fixed the same drift in `device-smoke.sh` (`CYAN: unbound variable`). Fix in `deploy.sh`: drop colors entirely (output flows through the install log file, ANSI escapes are noise there). Fix in `status.sh`: replaced the bare `[[ -t 1 ]] || { GREEN=''; ‚Ä¶ }` non-TTY-only guard with a self-contained TTY-or-empty block that defines the legacy names locally ‚Äî no dependency on `unified-log.sh` internals from interactive scripts going forward.
- **`scripts/diagnostic.sh` now anonymises `hw.txt`** ‚Äî the diagnostic bundle's `hw.txt` capture (`/proc/cmdline` + `/proc/cpuinfo` + `/etc/asound.conf` + overlayroot/df state) was written to the staging directory without passing through the `anonymise` filter, so MAC addresses injected by the Pi firmware into the cmdline (`smsc95xx.macaddr=<MAC>` on Pi 3 / 4) leaked into every install-failed bundle the user attached to a GitHub issue. The journal / docker / dmesg streams were already filtered. Now `hw.txt` is piped through `anonymise` on write ‚Äî same redaction rules (MAC, RFC1918 IPv4, SSIDs, credential lines, Bearer tokens). Found while inspecting the first real-world install-failed bundle generated by the PR #380 trap on snapvideo.
- **`scripts/diagnostic.sh` now captures `/var/log/snapmulti-install.log` in the bundle** ‚Äî the install log is the only place where the per-module INFO / WARN / ERROR lines emitted by `firstboot.sh` via `unified-log.sh` land. `journalctl` only captures the systemd-unit envelope (cloud-init / snapmulti-firstboot.service stdout/stderr) ‚Äî not the structured per-step log, which is where every interesting failure-mode signature lives (verify-server timeout, setup-script abort, mount-music failure, missing HAT detection). The first real-world install-failed bundle from snapvideo had every stream EMPTY except smoke.json ‚Äî `journal.log`, `docker.log`, `smoke.stderr`, `dmesg.log` all blank or trivial ‚Äî making root-cause analysis impossible. The new `install.log` capture is anonymised through the same filter and tail-limited to 5000 lines (covers a full install run with verbose deploy output). Tested locally: bundle from a deploy.sh dry-run includes the expected per-module timestamps and module tags.
- **`cmdline_ensure_memory_cgroup` removes `cgroup_disable=memory` before appending the enable token** ‚Äî the previous implementation relied on kernel "last-wins" semantics when both `cgroup_disable=memory` and `cgroup_enable=memory cgroup_memory=1` were present in `/proc/cmdline`. That assumption broke in the field on snapvideo (Pi 4 8GB) ‚Äî the memory cgroup hierarchy was absent at runtime even though both tokens were in cmdline.txt, so Docker's `mem_limit` settings in compose were silently ignored and every container reported `HostConfig.Memory=0`. The helper now strips the disable token first via the existing field-based `cmdline_remove_token`, leaving only the explicit enable. Idempotent.
- **`deploy.sh` first compose start now goes through `systemctl start snapmulti-server.service`** ‚Äî previously `install_systemd_service` ran AFTER `start_services` had already done a raw `docker compose up -d`. The systemd unit was therefore registered but never invoked on its own ExecStart path during firstboot, so the `ExecStartPost` mDNS self-heal (12 s wait + PTR-only-publish detection + snapserver restart) and the `PartOf=avahi-daemon.service` binding only kicked in on later restarts. Result: every fresh install was vulnerable to the libavahi-client startup race that caused PTR-only mDNS publication on first boot. Fix: install the unit first, then have `start_services` hand off to `systemctl start snapmulti-server.service` when the unit is registered (raw `docker compose up -d` is still the fallback for `deploy.sh` runs from a dev host without the unit installed).
- **`mount-music.sh` aborts NFS / SMB setup when `systemctl enable .automount` fails** ‚Äî the previous `if ‚Ä¶ ; then OK ; else log_warn ; fi` left the function returning 0 even when the automount could not be enabled, so an install completed "successfully" with the `.mount` / `.automount` files on disk but no systemd registration. The library was then silently unreachable at every boot. The helper now returns non-zero on enable failure and the NFS / SMB branches in `setup_music_source` propagate the abort, surfacing the underlying daemon-reload / enable problem at install time instead of at the user's first attempt to play music.
- **`firstboot.sh` SKIP_DOCKER verify accepts `enabled + waiting-for-reboot` on Pi Zero 2W** ‚Äî `setup-zero2w.sh` writes `dtoverlay=$HAT` to `/boot/firmware/config.txt` and best-effort loads it at runtime via `dtoverlay`. When the runtime load fails (common: bootloader-level overlays only take effect after the post-firstboot reboot), `snapclient.service` is enabled with the correct `--soundcard` but cannot become active until the kernel registers the HAT. The previous `systemctl is-active --quiet` gate aborted firstboot with `exit 1`, looping the install. The check now accepts `enabled` as a valid intermediate state, logs a warning + last 10 journal lines for context, and lets firstboot finalise; the HAT comes up after reboot and the post-reboot smoke gate catches the case where the HAT genuinely never registers.
- **`docs/USAGE.md` Tidal Connect description** ‚Äî "Exception 2" line was self-contradictory ("opt-in" in the same sentence as "opt-out by default"). Reworded to consistently describe the `tidal` Compose profile as opt-in. Italian mirror synced.

### Fixed
- **`snapmulti-boot-tune.service` failed at boot on WiFi-only Pi** ‚Äî `scripts/boot-tune.sh:35` ran `eth_ip=$(ip -4 addr show eth0 2>/dev/null | awk '/inet /{print $2; exit}')` to detect whether ethernet had carrier (so WiFi could be disabled to avoid dual mDNS publication). Under `set -euo pipefail` the `$()` substitution kills the whole script in ~80 ms ‚Äî the systemd unit reports `Failed with result 'exit-code'` and **none** of the later tuning runs: no `bcm2835_wdt` modprobe (watchdog disabled), no `cgroup_enable=memory` cmdline path (Docker memory limits silently ignored), no `/media/root-rw` tmpfs resize on overlayroot (defaults to 10% of RAM ‚Üí ENOSPC under apt install), no CAKE qdisc + DSCP EF marking on server egress (bufferbloat desyncs rooms), no per-port DSCP rules. PR #334 generated a `.automount` companion unit with `After=network-online.target` + `Wants=network-online.target`. Those directives created a cycle: `sysinit.target ‚Üí local-fs.target ‚Üí media-nfs\x2dmusic.automount ‚Üí network-online.target ‚Üí network.target ‚Üí ... ‚Üí sysinit.target`. systemd resolved it by *deleting* `local-fs.target`, `sockets.target`, `systemd-update-done.service`, and `systemd-binfmt.service` from the boot ‚Äî `journalctl -b 0 | grep "ordering cycle"` showed 10+ such cycles on every reflashed device. SIGTERM from `docker stop` (issued by `snapclient.service` / `snapmulti-server.service` `ExecStop=docker compose down`) was queued forever; Docker waited `stop_grace_period` (default 10 s) then SIGKILLed, but the host-visible PID lingered as a zombie while `systemd-shutdown` waited `DefaultTimeoutStopSec` (~90 s) per container. On a `both` install (server + client on the same Pi) that meant fb-display + audio-visualizer + metadata-service √ó 90 s ‚âà 4.5 minutes of stall per reboot. Three coordinated fixes: (1) all three Python files moved their signal registration into a small `_async_main()` wrapper that calls `loop.add_signal_handler(SIGTERM, cleanup, ...)` so the signal is processed by the asyncio loop directly; (2) the corresponding compose entries (`client/common/docker-compose.yml` for fb-display + audio-visualizer, `docker-compose.yml` for metadata) gained `init: true` (tini as PID 1 to propagate signals + reap zombies) and `stop_grace_period: 2s` (Docker SIGKILLs fast so the host-visible PID disappears before systemd-shutdown enumerates it); (3) `metadata-service.py` was bind-mounted into its container ‚Äî same pattern as the existing fb-display + visualizer mounts ‚Äî so the Python fix propagates without an image rebuild
- **First boot post-install dropped to emergency mode with "Press Enter to continue / Cannot open access to console"** ‚Äî the overlayroot initramfs hook (with `recurse=0`) rewrites `/etc/fstab` during the first boot after `setup_readonly_fs` enables the read-only root: `/media/nfs-music` is mapped to `/media/root-ro/media/nfs-music` AND the `nofail` flag is stripped along the way. When the NAS was slow on the very first attempt (cold ARP/DNS, NFS server still booting), systemd-fstab-generator emitted a `media-root\x2dro-media\x2dnfs\x2dmusic.mount` unit treated as a hard `local-fs.target` dependency. The mount failed, `local-fs.target` failed, ordering cycles surfaced for `NetworkManager.service` / `systemd-update-done.service` / `local-fs.target`, and systemd landed in `emergency.target`. Two coordinated fixes: (1) `mount-music.sh` no longer writes `/etc/fstab` for NFS/SMB shares ‚Äî it generates a hand-crafted `.mount` unit in `/etc/systemd/system/` (which the overlayroot hook does NOT rewrite). The unit name is built via `systemd-escape -p --suffix=mount`, the options include `nofail`, and `Before=snapmulti-server.service snapclient.service` orders the mount BEFORE the snapMULTI services without making them `Requires=` (so a slow NAS still cannot route the boot into emergency mode); (2) `setup_readonly_fs` now runs `update-initramfs -u -k all` explicitly after `raspi-config nonint do_overlayfs 0` (raspi-config rebuilds the initramfs itself, but if a kernel-module apt install raced with raspi-config the rebuild can lag ‚Äî the explicit follow-up is cheap insurance) and sanity-checks that `cmdline.txt` carries `console=tty1` so a future emergency-mode failure is at least *visible* on the HDMI console rather than rendered to a sealed serial port
- **mDNS published PTR-only records on first boot ‚Äî strict clients failed to discover Snapcast** ‚Äî `snapmulti-server.service` and `snapclient.service` only had a `docker info`-readiness `ExecStartPre`. When `docker info` returned ready before avahi-daemon had finished initialising, snapserver's `libavahi-client` connect race-lost: the connection silently degraded to PTR-only via raw UDP 5353 multicast, and the SRV/TXT follow-up queries that strict mDNS clients send (Python `zeroconf`, `dns-sd`, snapclient 0.36+) were never answered. The same race hit snapclient's discovery (`snapclient-discover` runs `avahi-browse` against an empty cache). Symptom in `device-smoke.sh`: recurring `[ERROR] Snapcast mDNS publishes PTR but NO SRV+TXT ‚Äî strict clients fail. Try \`docker compose restart snapserver\``. Two coordinated fixes: (1) both units gained a second `ExecStartPre` that polls `systemctl is-active --quiet avahi-daemon.service` AND `[[ -S /run/avahi-daemon/socket ]]` for up to 30 s, then `sleep 2` for avahi's first announce ‚Äî non-fatal so the unit still comes up on systems without avahi; (2) `snapmulti-server.service` gained an `ExecStartPost=-` that, 12 s after `compose up`, queries `avahi-browse -prt _snapcast._tcp` for the local cache (NOT `-prtl` ‚Äî the `-l` flag is `--ignore-local` which would exclude services published by THIS host's avahi-daemon, defeating the check) and restarts the `snapserver` container if it sees a `+;‚Ä¶;_snapcast._tcp` PTR but NO `=;‚Ä¶;_snapcast._tcp` SRV/TXT pair. The leading `-` makes the hook ignore-failure so a transient browse error never holds the unit in failed state. Pairs with the existing `PartOf=avahi-daemon.service` (PR #300) which still propagates LATER avahi restarts; this fix closes the FIRST-BOOT race
- **Install TUI overlapped fb-display on the HDMI console** ‚Äî both the `firstboot.sh` install progress (rendered via fbcon on `/dev/tty1`) and the fb-display container (raw pixels on `/dev/fb0`) were fighting for the same framebuffer surface. The overlap was visible at the end of the client install (firstboot calls `systemctl start snapclient.service` BEFORE the reboot, so fb-display starts while the install TUI is still drawing) and again post-reboot, where the default `getty@tty1.service` login prompt would redraw on top of fb-display. Three coordinated fixes: (1) `progress.sh` and `unified-log.sh` now write to `$PROGRESS_TTY` (default `/dev/tty1` for backward-compat with standalone runs); (2) `firstboot.sh` exports `PROGRESS_TTY=/dev/tty3` + `chvt 3` at start so the install TUI gets its own VT, then `chvt 8` (a blank VT outside `logind`'s autovt range) right before starting `snapclient.service` so the kernel `fbcon` driver clears the framebuffer for fb-display to draw on without contention; (3) client `setup.sh` now masks `getty@tty1.service` when `HAS_DISPLAY=true` ‚Äî console login remains accessible via Alt+F2 (logind autovt template auto-spawns agetty on tty2 on demand)

### Fixed

Upstream fix would require changing `publish_avahi.cpp::client_callback` to retry on `AVAHI_CLIENT_FAILURE` instead of quitting ‚Äî tracked separately
- **fuse-overlayfs pre-pull switch is now checkpointed** ‚Äî the pre-configuration of Docker's storage driver in `firstboot.sh` runs `rm -rf /var/lib/docker/*` to drop stale overlay2 layers, but this destructive step had no checkpoint guard. Power loss between the wipe and the subsequent successful pull would re-run the wipe on next boot, clobbering valid images already pulled under fuse-overlayfs. Added `checkpoint_reached "fuse-overlayfs-switched"` guard, set ONLY after the new daemon comes up and `docker info` confirms the driver actually flipped to fuse-overlayfs (10 iterations √ó 2 s = 20 s wait window). An interrupted switch correctly retries from scratch on next boot; a successful switch is never repeated. Closes the H9 follow-up from the Council 1 audit (left as the only Tier 1 deferred item in PR #298 due to file overlap with #297)

- **`snapmulti-docker-driver.service` repeated the same anti-pattern as the overlayroot cycle bug** ‚Äî the unit had `Before=docker.service` + `After=local-fs.target` + `WantedBy=multi-user.target` without `DefaultDependencies=no`. Same shape as the overlayroot service that was just neutralised by `recurse=0`. Pre-emptively added `DefaultDependencies=no` plus the standard `Conflicts=shutdown.target` / `Before=shutdown.target` pair, so this oneshot is fully insulated from `multi-user.target`'s implicit ordering and cannot be the next "36 cycles on first boot" candidate
- **Checkpoint files could be marked done while truncated** ‚Äî `firstboot.sh` used `touch` + `[[ -f ]]`, so a power loss between `touch` and the step's actual completion would leave a zero-byte file that `checkpoint_reached` accepted as success, silently skipping the step on next boot. Switched to atomic write (`.tmp` + `sync` + `mv`) and `[[ -s ]]` (non-empty required). Also writes a UTC timestamp into the checkpoint so post-mortem inspection shows when each step actually completed
- **Overlayroot ordering cycles on first post-firstboot boot** ‚Äî `firstboot.sh` wrote `/etc/overlayroot.local.conf` with `overlayroot="tmpfs"` (default `recurse=1`). overlayroot then wrapped every other fstab entry (NFS, USB) under `/media/root-ro/`, creating systemd ordering cycles between `media-root\x2dro-media-nfs\x2dmusic.mount`, `remote-fs-pre.target`, `local-fs.target` and the synthesized overlay redirect. Observed live on pi-server: 36 ordering cycles on the first boot post-firstboot, ~4 minutes of degraded state, manual reboot needed before the second boot resolved (timing-dependent recovery ‚Äî same fstab, no cycles). Switched to `overlayroot="tmpfs:recurse=0"` so only `/` is overlayed; NFS/USB entries stay at their natural mount points. Same change in `client/common/scripts/ro-mode.sh` so `ro-mode enable` produces the same config
- **Compact TUI installation display** (`9a4d83e`) ‚Äî shortened the elapsed line and expanded the log area to 14 lines so log + progress + checklist all fit on the 800√ó600 framebuffer
- **I2C HAT detection false positive on bare Pi 4/5** ([#276](https://github.com/lollonet/snapMULTI/pull/276)) ‚Äî scan restricted to bus 1 (GPIO header); HDMI internal buses (20/21 on Pi 4, 11/12 on Pi 5) had devices at PCM5122 addresses causing wrong HiFiBerry detection, ALSA crash, and install failure
- **NFS mount blocking firstboot on slow networks** ([#275](https://github.com/lollonet/snapMULTI/pull/275)) ‚Äî `mount-music.sh` now writes the fstab entry before attempting the mount and runs the mount with `timeout 30`; SMB credentials retained on failed mount so systemd retries with `nofail` succeed on the next boot
- **Headless clients opened wrong ALSA card** (`b8a4b58`) ‚Äî snapclient 0.35.0 internally translates bare `default` to `sysdefault`, opening card 0 (HDMI on Pi Zero 2W ‚Üí ALSA error 524 with no display attached). Headless `SOUNDCARD` now uses an explicit `hw:` form, bypassing the translation
- **MPD verify timeout too tight during firstboot (60s)** ([#279](https://github.com/lollonet/snapMULTI/pull/279)) ‚Äî `deploy.sh verify_services` gave up after 60s when `MPD_START_PERIOD=30s`, but Pi 4 2GB during firstboot needs 90-120s for MPD's listener bind due to cold caches and post-pull I/O contention. Floor verify timeout at 120s, decoupled from MPD's healthcheck `start_period`. Network mounts (NFS/SMB) still get the extended 300s+ window
- **MPD verify timeout still too tight on Pi 4 2GB (120s)** ([#281](https://github.com/lollonet/snapMULTI/pull/281)) ‚Äî bumped `verify_services` floor from 120s to 180s after live observation on pi4-2gb-srv `--both` install (MPD needed 130-150s for listener bind)
- **Snapclient hardware mixer "Digital" not found on headless HiFiBerry** ([#282](https://github.com/lollonet/snapMULTI/pull/282)) ‚Äî snapcast 0.35.0's `alsa_player.cpp` parses the soundcard name to derive the mixer card. With `hw:NAME,0` it splits on `:` and ends up with `mixer_device_ = "hw"` (broken), so `snd_mixer_attach("hw")` finds no controls and snapclient crashes with "Failed to find mixer: Digital". Switched headless `SOUNDCARD` to the equivalent `hw:CARD=NAME,DEV=0` form, which contains the literal `CARD=` substring snapcast needs
- **`device-smoke.sh` autodetect misclassified client-only as both** ([#283](https://github.com/lollonet/snapMULTI/pull/283)) ‚Äî `detect_dir` fell back to `${SCRIPT_DIR}/..` when `/opt/snapmulti` was missing. On client-only installs the script lives at `/opt/snapclient/scripts/`, so the parent (`/opt/snapclient`) matched as a "server" because it had `docker-compose.yml + .env`. Now `detect_dir` validates the candidate compose actually declares the role's service (`snapserver:` or `snapclient:`)
- **Snapclient failover never triggered when its server died** ([#285](https://github.com/lollonet/snapMULTI/pull/285)) ‚Äî `discover-server.sh --watch` updated `SNAPSERVER_HOST=` in `.env` then ran `docker compose restart`, but `restart` does not re-read `.env`, so snapclient came back up bound to the old (dead) IP and stayed there. Switched to `docker compose up -d` (re-evaluates compose+env, recreates only when effective config differs). Combined with five companion changes to give real multi-server failover:
  - TCP-probe the current `SNAPSERVER_HOST` on port 1704 before scanning mDNS ‚Äî no flapping when the current server is alive but mDNS happens to return another one first
  - Enumerate *all* mDNS results and prefer one different from the (dead) current ‚Äî avoids getting stuck on stale Avahi cache (1-3 min TTL)
  - Tightened `snapclient-discover.timer` from 5 min to 60 s ‚Äî failover within ~1 min instead of up to 7 min worst-case
  - Reconcile drift between container env and `.env`: recreate via `up -d` if `docker inspect` shows a different `SNAPSERVER_HOST` than `.env` (heals devices left in env-drift state by the legacy `restart` bug)
  - Reconcile container runtime state: also `up -d` when `docker inspect`'s `State.Running == false` ‚Äî covers transient docker glitches that leave the container in `Created` state with correct config but never started
  - Added `[discover]` log prefix and mDNS result list to `journalctl -u snapclient-discover.service` for diagnosis
- **Empty `/etc/resolv.conf` after boot on Pi Zero 2W / low-RAM boards** ([#287](https://github.com/lollonet/snapMULTI/pull/287)) ‚Äî at boot, NetworkManager attempts an atomic write of `/etc/resolv.conf` (tmpfile + rename). Under tmpfs pressure on 512MB boards, this can fail with `write() failed: No space left on device` (transient ENOSPC despite ample free space). NM gives up after one failure and never retries ‚Äî its DNS state is correct internally (`nmcli ... IP4.DNS` shows the real server) but `/etc/resolv.conf` stays empty. `wait-network.sh` stage 3 now first calls `nmcli general reload dns-rc` to force NM to re-commit its known DNS, recovering the real network DNS instead of falling back to public 1.1.1.1
- **Snapserver mDNS announcement was incomplete (PTR only, no SRV/TXT)** ([#290](https://github.com/lollonet/snapMULTI/pull/290)) ‚Äî the `snapserver` container did not bind-mount `/run/avahi-daemon/socket`, so libavahi-client couldn't reach the host's avahi-daemon. snapserver fell back to publishing PTR via raw multicast but never answered SRV/TXT follow-ups, so strict clients (e.g. Python `zeroconf`'s `get_service_info`, macOS `dns-sd -L`) returned None and reported "no servers found". Lenient clients (snapclient C++ binary, Pi `discover-server.sh`) usually worked because avahi-daemon on the same Pi cached `pi-server.local` from other services and synthesised the resolve, plus `.env` cemented `SNAPSERVER_HOST` after the first success. Adding the socket bind-mount (same pattern as the `mpd` service) lets snapserver register via libavahi-client; avahi-daemon now publishes PTR + SRV + TXT + A correctly for any client
- **Containerd Leases plugin self-heal at boot** ([#292](https://github.com/lollonet/snapMULTI/pull/292)) ‚Äî Pi 3 / Pi Zero 2W can hit transient ENOSPC at boot when containerd creates its metadata bolt DB (`/var/lib/containerd/io.containerd.metadata.v1.bolt/meta.db`); containerd starts but the `io.containerd.lease.v1` plugin stays unloaded. Every subsequent `docker run` / `compose up -d` then fails with `Unimplemented: unknown service containerd.services.leases.v1.Leases` and containers stay in `Created` state forever. Same false-ENOSPC family as the NetworkManager resolv.conf case fixed in [#287](https://github.com/lollonet/snapMULTI/pull/287). `boot-tune.sh` now greps the containerd journal for the specific error string and restarts the containerd + docker stack when found ‚Äî auto-recovery on next boot, no manual `systemctl restart` needed

### Added
- **`device-smoke.sh`: five new health checks** ‚Äî distilled from the live troubleshooting on 2026-05-07 where each of these conditions had a real-world manifestation in the same hour:
  - **NTP time sync** ‚Äî `timedatectl show -p NTPSynchronized` must be `yes`. Snapcast's TimeProvider is NTP-immune but log timestamps and metadata-service rely on a sane wall clock; Pi Zero 2W's RTC sits at epoch 0 until NTP completes, masking boot-time bugs in any component using absolute timestamps
  - **mDNS hostname round-trip** ‚Äî `avahi-resolve -n $(hostname).local` must return one of the host's own IPs (multi-homed Pis with both `eth0` and `wlan0` active are accepted). Catches DHCP/mDNS desync (e.g. IP changed via DHCP renew but avahi cache stale, or hostname mismatch)
  - **IP conflict detection** ‚Äî `arping -D` duplicate-address probe. Catches scenarios like the F8:17:2D PC HP statically configured with `.95` while pi-server also held `.95` via DHCP ‚Äî the kind of intermittent packet-loss bug that's invisible in steady state
  - **Snapcast mDNS strict-client publishing** (server-only) ‚Äî `avahi-browse -rpt _snapcast._tcp` must return at least one `=` (resolved with SRV+TXT) line; PTR-only fails the gate. Catches the upstream Snapcast 0.35.0 `AVAHI_CLIENT_FAILURE` bug that PR #290 partially fixed and PR #300 mitigated via systemd `PartOf=`
  - **Hardware throttle / under-voltage** ‚Äî `vcgencmd get_throttled`. Currently-active bits (0x1/0x4/0x8) hard-fail; sticky historical bits (0x10000/0x40000/0x80000) warn loudly. Under-voltage from a marginal PSU/cable is the #1 hardware-side support issue on community Pi installations
- **`iputils-arping` added to `install-deps.sh`** ‚Äî required by the new IP-conflict check; gated only on `arping` not already on PATH so existing installs are not over-installed
- **`device-smoke.sh`: seven Tier-2 health checks** ‚Äî round of follow-ups to the Tier-1 work, each motivated by a class of issues we want surfaced at install time (instead of weeks later as a "things suddenly stopped working" support ticket):
  - **AirPlay mDNS strict-client** (`_raop._tcp`) and **Spotify mDNS strict-client** (`_spotify-connect._tcp`) (server-only) ‚Äî same `avahi-browse -rpt` SRV+TXT verification as the Snapcast check from Tier 1, since shairport-sync and librespot share the same upstream avahi-client lifecycle bug
  - **Avahi socket bind-mount per container** ‚Äî verifies that PR #290 (snapserver) is still in `docker inspect` for the running container, plus mpd and snapclient. A redeploy from an older docker-compose.yml would silently lose the mount and degrade mDNS to PTR-only without this gate
  - **HAT consistency** (client/both) ‚Äî the expected ALSA card is resolved from `SOUNDCARD=hw:CARD=NAME,DEV=0` (headless format introduced by PR #282) OR from `/etc/asound.conf` (`ctl.!default { card NAME }` first, fallback to the first non-Loopback `hw:NAME` reference) when `SOUNDCARD=default` (display+HAT setups using the `multi` ALSA plugin to fan audio to both the HAT and a Loopback for audio-visualizer / fb-display). The resolved name must appear in `aplay -l`. Catches stale or mis-detected HAT configs (e.g. user swapped from HiFiBerry to InnoMaker without re-running setup.sh) and verifies that display+HAT routing didn't lose its DAC binding
  - **Audio path active-params** (client/both, when audio is playing) ‚Äî `/proc/asound/.../hw_params` must show `44100 / S16_LE / 2` when active. INFO when idle (no audio currently flowing ‚Äî not a fail), pass when active and at the snapMULTI invariant format, fail only when active but at the WRONG format
  - **Containerd self-heal counter** ‚Äî `/var/lib/snapmulti-installer/containerd-heal.count` (set by the H5 fix in PR #298). Above 3 ‚Üí fail (structural ENOSPC, not transient). 1-2 ‚Üí warn. 0 / absent ‚Üí pass. Gives operational visibility into the false-ENOSPC family on Pi Zero 2W / Pi 3
  - **Music library non-empty** (server-only, when `MUSIC_SOURCE=nfs|smb`) ‚Äî `find` for known audio file extensions under `MUSIC_PATH`. Local sources (USB, internal disk) deliberately skipped ‚Äî too many legitimate states (empty drive, fresh install, etc.). Network-backed shares are the case where empty = clear configuration error
  - **Snapcast JSON-RPC API** (server-only) ‚Äî `Server.GetStatus` POST to `http://127.0.0.1:1780/jsonrpc` must return 200 with a JSON body containing `"result"`. Smallest functional check that proves snapserver is not just `Up` per docker but actually serving control requests (e.g. for the Android Snapcast app, or our own discover-server.sh)

### Features

  - MPD (local music library)

  - Spotify Connect (via librespot)

- **Security hardening** ‚Äî read_only containers, no-new-privileges, cap_drop ALL, tmpfs mounts
- **Hardware auto-detection** ‚Äî Auto-selects resource profile (minimal/standard/performance) based on RAM and Pi model
- **Zero-touch SD preparation** ‚Äî `scripts/prepare-sd.sh` for automatic first-boot installation
- **Deploy script** ‚Äî `scripts/deploy.sh` bootstraps fresh Linux machines
- **Multi-arch images** ‚Äî amd64 + arm64 on ghcr.io
- **Bilingual docs** ‚Äî English + Italian

### Technical

- Audio format: 44100:16:2 (CD quality)
- Container architecture: 5 separate services communicating via named pipes
- CI/CD: GitHub Actions with self-hosted runners
- Resource profiles: minimal (Pi Zero 2/Pi 3), standard (Pi 4 2GB), performance (Pi 4 4GB+/Pi 5)

## [Unreleased]

### Fixed
- **install-deps: `apt-get upgrade` ‚Üí `apt-get dist-upgrade` so kernel + rpi-eeprom are not kept back** ‚Äî plain `upgrade` refuses to install new packages or remove old ones, so meta-packages like `linux-image-rpi-2712` / `linux-image-rpi-v8` / `rpi-eeprom` are silently held back when the new version requires installing a new sub-package (the new kernel image deb). Observed on snapvideo 2026-05-29: 6.12.75 ‚Üí 6.18.29 kernel + 28.15 ‚Üí 28.24 eeprom all stuck after a clean reflash. Fix applies to both `install-deps.sh` and the final upgrade pass in `firstboot.sh`. `apt-get dist-upgrade -y` is the standard for "actually apply pending kernel + eeprom updates".
- **install-deps: drop `dstat` from monitoring tools (pulls full `pcp` stack)** ‚Äî `dstat` Depends on `python3-pcp`, which transitively pulls Performance Co-Pilot (~50 MB, multi-daemon). The original code commented `--no-install-recommends` blocks `pcp` ‚Äî wrong, `pcp` is a HARD Depends. `iotop-c` ALSO Depends on libpcp3/libpcp-pmda3, so `pcp` still gets installed transiently and removed by the trailing purge; but at least we stop adding `dstat` on top. `sar` from sysstat covers dstat's main use case (CPU/mem/disk realtime).
- **install-deps: comment block corrected re. pcp Depends vs Recommends** ‚Äî the old comment claimed `--no-install-recommends` keeps pcp out; in reality `iotop-c` brings it in as a Depends. The comment now documents the transient install-then-purge correctly so future maintainers don't chase a non-bug.

## [0.7.9.5] ‚Äî 2026-05-29

> Script-only patch (image_set stays 0.7.7). Boot-time acoustic and smoke-page UX polishing on first-reflash + post-firstboot reboot: the tone now lands PASS on local / small NFS installs (#542) and the /status page no longer surfaces transient first-boot states as FAIL (#543). Validated on snapvideo with 76k-track NFS library.

### Fixed
- **smoke: two false-positive FAILs during first-boot window** ‚Äî
  - `check_boot_health.sh`: `systemd state unexpected: 'starting'` was fired when smoke was invoked by `snapmulti-status.timer` (no `SNAPMULTI_AUTO_BOOT` env var) during the first ~10 min after boot, while MPD was still scanning a large library and other services were still settling. The 'starting' tolerance now ALSO holds when uptime < 600 s ‚Äî bypass via env var or boot window, whichever applies.
  - `device-smoke.sh` `/status` snapshot check: PR #538 demoted "snapshot missing" to INFO only when uptime < 300 s AND the timer had never fired. Misses the case where the timer DID fire on schedule (OnBootSec=4min) but `snapmulti-status.service` is currently in ExecStartPre (waiting up to 5 min for MPD healthy on first boot) ‚Äî service state `activating`, snapshot legitimately not yet on disk. Now demotes to INFO when `systemctl is-active snapmulti-status.service` reports `activating`.

### Changed
- **auto-boot-smoke: mode-aware wait cap (server/both 300 s, client 90 s)** ‚Äî on server/both installs the boot-time smoke check now waits up to 300 s for snapserver / snapclient healthy (was 90 s); on client / client-native it stays at 90 s because snapclient is ready in seconds (no MPD, no library scan). The extension lets small / medium libraries (‚â§ ~10 k tracks local or NFS) finish MPD's first scan before the tone fires ‚Üí PASS chime instead of always FAIL during MPD warmup. Very large libraries (50 k+ NFS) still exceed 300 s ‚Üí fail tone remains, covered by the existing TROUBLESHOOTING entry + `backup-from-sd.sh` mpd.db pre-warm workflow. `TimeoutStartSec` bumped 240 ‚Üí 720 s only on the server/both unit (client keeps 240 s).

### Changed
- **fb-display: show "Booting‚Ä¶" instead of `?.?.?.?` during the first 90 s of boot (#540)** ‚Äî the bare placeholder reads as "network broken" on a fresh HDMI screen and triggered unnecessary user reboots during the normal WiFi-association + DHCP window. `_get_lan_ip()` now consults `/proc/uptime` after the socket probes fail: `< 90 s` ‚Üí `"Booting‚Ä¶"`, `‚â• 90 s` ‚Üí `"?.?.?.?"` (real persistent failure surfaces). `/proc/uptime` unreadable falls through conservatively to `"?.?.?.?"` so an outage is never silently hidden. The check now probes directly: (a) snapshot file missing on host ‚Äî INFO on fresh boot (< 5 min uptime + timer not yet armed), FAIL after; (b) file present on host but metadata container cannot read it ‚Äî FAIL pointing at the bind-mount + PUID/PGID; (c) file present and readable in container but service still returns 503 ‚Äî FAIL with the actual response body so the cause is visible. No false-positive on the first-boot window before `OnBootSec=4min`.

## [0.7.9.1] ‚Äî 2026-05-28

> Script-only patch (image_set stays 0.7.7). Resolves the v0.7.9 persistence regression. Field-validated on snapvideo: overlay active + `server.json` + `server.json.prev` + 4 groups all survived reboot under `overlayroot="tmpfs"`.
>
> **Known limitation:** very large NFS/SMB libraries (~76 k+ tracks observed) can still exceed the install healthcheck window even with the new 120 s stability grace introduced in this release. When that happens, `firstboot.sh` writes `.install-failed`, the `[finalize]` step never runs, and the device boots on plain ext4 (no overlay protection) ‚Äî even though containers are healthy and the LAN-side appears fine. Pre-set `MPD_START_PERIOD=3600s` in `install.conf` BEFORE flashing the SD card. See [docs/ADVANCED.md ‚Äî MPD_START_PERIOD](docs/ADVANCED.md#mpd_start_period) and [docs/TROUBLESHOOTING.md ‚Äî Install marked failed but containers run](docs/TROUBLESHOOTING.md#install-marked-failed-but-containers-run).

### Fixed
- **v0.7.9 persistence regression cascade (#527)** ‚Äî multiple compounding bugs from the v0.7.9 persistence work, all corrected here. The original `snapmulti-data-persistence.service` bind-mounted `/opt/snapmulti/data` over `/media/root-rw/snapmulti-persist/`, but `/media/root-rw/` is itself tmpfs on `overlayroot="tmpfs"` ‚Äî the bind worked at runtime but the persistent location was wiped at reboot, and the smoke check `[OK] Persistent bind active` actively masked the failure. Replaced with the proven `/boot/firmware/`-backup pattern (FAT32, the only userland-writable path that survives reboot on tmpfs overlayroot):
  - **Backup trigger** is split: `snapmulti-state-backup.path` watches myMPD workdir (event-driven, ~1 s latency for playlist/theme/script edits); `snapmulti-state-backup.timer` runs every 5 min for `server.json` (snapserver rewrites that file every ~3 s to refresh client `lastSeen` heartbeats ‚Äî wiring it to `.path` would cause `~354` backup-service invocations per 5 min and churn the FAT32 boot partition). `backup-snapmulti-state.sh` short-circuits on canonical equality (sort keys + strip `lastSeen` + sha256 compare) so a no-op timer tick costs only one JSON parse ‚Äî zero FAT32 writes when only heartbeats changed.
  - **Restore** runs as `ExecStartPre` on `snapmulti-server.service` BEFORE compose up, so snapserver finds restored state instead of writing a fresh empty one. The script PUID/PGID is read from `/opt/snapmulti/.env` for non-default user installs.
  - **Scope** matches the original bind-mount: snapserver `server.json` + WHOLE `myMPD/workdir/` (state + config + smartpls + scripts + pics). Narrowing to `state/` alone would silently lose user-customised smart playlists / theme / scripts on reboot.
  - **Seed on install** runs `snapmulti-state-backup.service` once when arming the watcher ‚Äî protects upgrade scenarios where state pre-exists the watcher and would otherwise leave the boot-partition backup stale until the next user change.
  - **Boot partition mount state preserved** ‚Äî `backup-snapmulti-state.sh` + `backup-mpd.sh` detect whether `/boot/firmware` was previously ro and restore THAT state on exit. The original v0.7.9 fix unconditionally remounted ro at exit, which during firstboot (both-mode) cascaded to break `raspi-config do_overlayfs` and prevented overlay activation entirely ‚Äî discovered by user pattern-recognition ("readonly in server mode has always worked").
  - Removed `snapmulti-data-persistence.service`, `snapmulti-data-setup.sh`, and the misleading `check_persistence.sh` smoke module; replaced with `check_state_backup.sh` (verifies backup freshness on the actual persistent path) + `check_timers.sh` now also asserts the path + timer units are enabled and active.
  - **`backup-from-sd.sh` catches up to widened backup scope (#530)** ‚Äî host-side reflash extractor now handles the current `mympd/workdir/` layout (atomic stage+swap to avoid merging stale files from the previous SD with the restored backup) and restores `data/server.json`. Legacy `mympd/state/`-only backups still work with a warning that user customisations aren't in the backup.
  - **Backup integrity + `.prev` rotation** ‚Äî `backup-snapmulti-state.sh` validates each `server.json` write (size ‚â• 64 B + JSON parse via `jq` when available) before publishing, and rotates the previous good copy to `server.json.prev` so a single transient corruption (truncated write, FAT32 fragmentation, power loss mid-publish) cannot brick the device. `restore-snapmulti-state.sh` validates `server.json`; falls back to `.prev` with a WARN log if current is corrupt; fails loud only when BOTH are unparseable. Plus a post-`verify_services` seed in `deploy.sh` so the boot-partition backup captures real state on first install (the in-place seed at install time was a no-op on fresh installs because snapserver hadn't written yet). 2 new bash tests (`test_backup_integrity_validation.sh`, `test_restore_prev_fallback.sh`) lock the lessons in CI.
- **`verify_services` healthcheck stability grace (120 s)** ‚Äî Docker does not necessarily mark a container healthy the instant the inner service binds its port; the next health probe runs up to 30 s later, and `verify_compose_stack` also needs a second sample to prove `RestartCount` stability. Large NFS libraries were observed to bind just after the 300 s start_period and then fail `verify_services` seconds before the next successful probe. New explicit `health_grace_seconds=120` on top of `max(MPD_START_PERIOD, 180s)` covers that window. The install log now exposes the full budget (`up to ${verify_budget}s, ${wait_seconds}s interval; MPD start_period=${start_period}s + 120s grace`). Does not eliminate the need to bump `MPD_START_PERIOD` for very large libraries ‚Äî see Known limitation above.
- **Mount smoke accepts active `.mount` as valid (`check_mounts.sh`)** ‚Äî some systemd versions mark the `.automount` companion inactive once the actual `.mount` has fired (e.g. after MPD scanned the library). Previously this scored as a fail. Now passes when either the automount is watching OR the mount has already fired; only fails when neither holds.

## [0.7.9] ‚Äî 2026-05-28

> Script-only patch (image_set stays 0.7.7). Network-stack simplification (IPv6 off at kernel level) + expanded status-page coverage. Resolves a class of dual-stack mDNS / Snapcast discovery failures.
>
> **Known issue (fixed in [0.7.9.1]):** `snapmulti-data-persistence.service` shipped in this version does NOT actually persist on the standard `overlayroot="tmpfs"` setup. The smoke check `[OK] Persistent bind active` is misleading. See #527 + the [0.7.9.1] entry above for the proper fix.

### Added (FAILED ‚Äî see Known issue)
- ~~**snapserver + myMPD state persistence (`snapmulti-data-persistence.service`, #525)**~~ ‚Äî architecturally broken on tmpfs-mode overlayroot. The bind-mount source `/media/root-rw/snapmulti-persist/` was itself tmpfs, wiped at reboot. Replaced in [0.7.9.1] with `/boot/firmware/` backup pattern.
- **Status page coverage expanded (#525)** ‚Äî 6 new checks surface previously-invisible state:
  - `/boot/firmware` FAT32 free space (fixed 512 MB partition ‚Äî full = no diagnostic bundles, no MPD backup, no kernel upgrade)
  - `/media/root-rw` backing fs free space (full = overlay cannot grow, OOM imminent)
  - Memory headroom + `MemAvailable` (would have caught the mympd OOM in v0.7.8.16 hours earlier ‚Äî surfaces 80 %+ pressure)
  - Swap pressure (heavy swapping degrades audio path latency)
  - Pending update (queries `/version` endpoint ‚Äî operator sees at a glance if a reflash brings a newer release)
  - Now-playing stream + track (which stream the connected clients actually hear, plus track metadata)
  - Previous boot end state (info-level ‚Äî flags kernel panic / power loss vs clean reboot)

### Changed
- **IPv6 disabled by default on every install (ADR-007, #522)** ‚Äî `prepare-sd.sh` / `prepare-sd.ps1` now write `ipv6.disable=1` to `/boot/firmware/cmdline.txt` by default. snapMULTI is a LAN-only audio appliance and dual-stack mDNS / Snapcast discovery causes intermittent silent failures on consumer LANs (see #521 + the broader Avahi dual-publish race in #425). Kernel-level disable pre-empts every userland subsystem, lives outside the overlayroot upper layer, and is reversible by removing the token from `cmdline.txt`. Opt out with `DISABLE_IPV6=false ./scripts/prepare-sd.sh ...` for the rare operator with a hard IPv6 requirement. Documented in [ADVANCED](docs/ADVANCED.md#ipv6-disabled-by-default) + [TROUBLESHOOTING](docs/TROUBLESHOOTING.md#ipv6-disabled--is-this-a-problem). `install-deps.sh` installs `Acquire::ForceIPv4 "true"` apt config so first-boot package fetches don't stall on AAAA timeouts.

### Fixed
- **Pi Zero 2W ‚Äî snapclient IPv6 link-local latch (#521 ‚Üí #522)** ‚Äî snapclient's built-in libavahi-client browse occasionally latched onto an IPv6 link-local SRV target that did not route on the local LAN, leaving audio silent with a healthy server visible to every other client. #521 shipped a surgical workaround (`ExecStartPre` pin `--host <IPv4>` into `/etc/default/snapclient`). #522 supersedes it by killing IPv6 at the kernel level ‚Äî the workaround is removed, snapclient's built-in mDNS auto-rediscovery is restored for free, and the same class of bug stops affecting other consumers (fb-display, future zeroconf code).
- **mympd crash-loop after ADR-007** ‚Äî `ghcr.io/jcorporation/mympd` upstream default is `MYMPD_HTTP_HOST=[::]:8180` (dual-stack listen). With IPv6 disabled at kernel level the bind returns `EAFNOSUPPORT` (errno 97) and the container exits, re-spawning every few seconds. Pin `MYMPD_HTTP_HOST=0.0.0.0` in `docker-compose.yml`. Caught immediately after rolling ADR-007 to snapvideo. snapserver / MPD / metadata-service already bind IPv4 explicitly ‚Äî mympd was the only consumer of the upstream IPv6 default.

### Removed
- **PR #521 workaround retired (#522)** ‚Äî `discover-server-native.sh` + the `ExecStartPre=+/usr/local/sbin/snapmulti-discover-snapserver` line on the Pi Zero 2W drop-in. Native auto-rediscovery restored.

### Fixed

### Changed
- **Smoke classifies past container restarts and paired-offline Snapcast clients as info, not failure** ‚Äî observed live on the fleet: `[ERROR] Container(s) with RestartCount > 0 (crash-loop): snapclient(RC=1)` was emitted on every smoke run for nodes that had crashed once during firstboot and recovered cleanly. RestartCount is a lifetime counter, so a one-shot crash that healed was indistinguishable from an active crash-loop. Same noise on the Snapcast side: paired clients that had been offline since the last reboot surfaced as `WARN: 2/3 connected (1 disconnected)`, which is just stale state, not a regression. Reclassified: `RestartCount > 0` with current state `running`/healthy ‚Üí `info` ("past restart observed"), only `restarting`/`dead`/`exited` or `unhealthy` remain `fail`. Paired-offline Snapcast clients ‚Üí `pass_check` + `info` with the offline client list, no more `warn`. `paused`/`created` deliberately not in the failure set (`paused` = operator intent, `created` + RC>0 is unreachable). New test: `tests/test_smoke_runtime_classification.sh` (3 scenarios ‚Äî Snapcast PASS with offline clients listed as info, container historical restart classified as OK + info, container unhealthy with RC>0 remains active failure).
- **`fleet-smoke.sh` separates non-snapMULTI Snapcast clients from SSH-managed fleet nodes** ‚Äî when an iOS / macOS Snapcast app connects to the server, it shows up in `_snapcast._tcp` as a connected client but is not a fleet node we SSH-smoke. Now the JSON-RPC parser splits clients by `host.os`: only `Debian GNU/Linux` (or empty `host.os` for pre-0.35 snapclient builds) goes into the SSH list; the rest are reported as `Connected non-snapMULTI clients: <name>(<ip>, <os>)` in the human output and as a top-level `connected_non_snapmulti_clients` array in the JSON output. (b) `scripts/device-smoke.sh` `_net_check_arping` no longer uses `arping -D`; it now sends a normal ARP request (`arping -c 3 -w 5`) and inspects reply MACs ‚Äî any reply from a MAC other than ours is the conflict signature, without the destructive tentative-address transition. New smoke check (see Added) catches future regressions.

## [0.7.1] ‚Äî 2026-05-10

Bug-fix release on top of v0.7.0. Two boot-time failures surfaced during
the v0.7.0 reflash gate (pi-server, pi-display, pi-zero) ‚Äî neither
preventable from the smoke output v0.7.0 shipped with. Bundled with a
comprehensive `device-smoke.sh` rewrite that makes both regressions
observable so a similar class cannot pass silently again. Validated on
5 devices: pi-server (Pi 4, --both), pi-display (Pi 4 2GB + display, --client),
pi3-1gb (Pi 3 1GB, --client), pi4-2gb-server (Pi 4 2GB + DAC HAT + USB, --both),
pi-zero (Pi Zero 2W 512 MB, --client).

### Fixed
- **`snapmulti-boot-tune.service` failed at boot on WiFi-only Pi** ‚Äî `scripts/boot-tune.sh:35` ran `eth_ip=$(ip -4 addr show eth0 2>/dev/null | awk '/inet /{print $2; exit}')` to detect whether ethernet had carrier (so WiFi could be disabled to avoid dual mDNS publication). On a Pi without an `eth0` device (Pi Zero 2W, any Pi 3/4 booted with ethernet cable unplugged), `ip -4 addr show eth0` exits 1 before `awk` ever runs. Under `set -euo pipefail` the `$()` substitution kills the whole script in ~80 ms ‚Äî the systemd unit reports `Failed with result 'exit-code'` and **none** of the later tuning runs: no `bcm2835_wdt` modprobe (watchdog disabled), no `cgroup_enable=memory` cmdline path (Docker memory limits silently ignored), no `/media/root-rw` tmpfs resize on overlayroot (defaults to 10% of RAM ‚Üí ENOSPC under apt install), no CAKE qdisc + DSCP EF marking on server egress (bufferbloat desyncs rooms), no per-port DSCP rules. The failure was invisible in v0.7.0 because the smoke had no boot-graph integrity check. Caught on pi-zero post-reflash 2026-05-10: `Process: 1350 ExecStart=/usr/local/bin/snapmulti-boot-tune.sh (code=exited, status=1/FAILURE)`. Fix: `eth_ip=$(... | awk ... || true)`. Confirmed live on pi-zero: smoke went from `1 unexpected failed unit(s): snapmulti-boot-tune.service` to all-green after deploying the patched script
- **First-boot post-overlayroot came up degraded ‚Äî `.automount` unit created a systemd ordering cycle that deleted `local-fs.target` and `sockets.target`** ‚Äî root cause of the "device came up without network, had to power-cycle" symptom on pi-server (Pi 4, server+client) and pi-display (Pi 4 2GB, client) after the v0.7.0 reflash on 2026-05-10. PR #334 generated a `.automount` companion unit with `After=network-online.target` + `Wants=network-online.target`. Those directives created a cycle: `sysinit.target ‚Üí local-fs.target ‚Üí media-nfs\x2dmusic.automount ‚Üí network-online.target ‚Üí network.target ‚Üí ... ‚Üí sysinit.target`. systemd resolved it by *deleting* `local-fs.target`, `sockets.target`, `systemd-update-done.service`, and `systemd-binfmt.service` from the boot ‚Äî `journalctl -b 0 | grep "ordering cycle"` showed 10+ such cycles on every reflashed device. On a minimal headless device (pi3-1gb, Pi 3 1GB) the degrade was tolerable; on heavier configurations (pi-server+pi-display with display, multiple containers, NFS) it surfaced as "no network at first boot". Fix: drop `After=network-online.target` and `Wants=network-online.target` from the `.automount` template. Pattern matches systemd-fstab-generator: ordering only on the `.mount` unit (which still has it), never on the `.automount` (which is just a watch on a directory entry). Test `test_mount_music_systemd_unit.sh` extended with an awk-based heredoc parser that asserts the `.automount` block carries no `network-online.target` / `nss-lookup.target` reference
- **First-boot post-overlayroot WARN noise ‚Äî `update-initramfs -u -k all` after `do_overlayfs 0` always fails** ‚Äî `scripts/common/system-tune.sh` had an explicit `update-initramfs -u -k all` after `raspi-config nonint do_overlayfs 0` (added in PR #317 as "cheap insurance"), but by that point raspi-config has already installed the `overlayroot` package and its initramfs hooks are partially live ‚Äî `mkinitramfs` then aborts with `failed to determine device for /` because it sees the overlay instead of the underlying ext4. The fail was silent (`>/dev/null 2>&1`) AND raspi-config's own `update-initramfs -c -k all` had already produced a working initramfs on disk, so this finding was cosmetic ‚Äî the WARN ("first boot may need manual reboot") was misleading. Fix: remove the extra `update-initramfs -u -k all` block; trust raspi-config's internal `-c -k all`. If a future race re-emerges the right path is to capture mkinitramfs output and apply MODULES=most ‚Äî NOT a silent extra rebuild. Test `test_mount_music_systemd_unit.sh` extended with a regression guard so the extra rebuild cannot come back

### Added
- **`device-smoke.sh` modular system checks (`scripts/smoke/check_*.sh`)** ‚Äî 6 new modules sourced by the release-gate smoke, each independently testable:
  - **`check_boot_health.sh`** ‚Äî counts `ordering cycle` lines in `journalctl -b 0` (the `.automount` regression above would have been a same-day fail with this in place) AND counts `Job *.service/start deleted` lines (the units systemd ripped out to break the cycle). Also runs `systemctl is-system-running` and filters the failed-unit list against a small allowlist (`rpi-resize-swap-file.service` legitimately fails on RO FS, `cloud-init-main.service` typically reports "failed" on completed firstboot due to runcmd timeouts) ‚Äî only flags UNEXPECTED failed units so `degraded` isn't a permanent false-positive
  - **`check_mounts.sh`** ‚Äî for `MUSIC_SOURCE=nfs/smb/network`, asserts the `.automount` unit is enabled (lazy path) and the `.mount` is NOT eager-enabled, then **reads the actual unit-file content** and greps the `.automount` for `(After|Wants|Requires|BindsTo)=.*network-online\.target` ‚Äî the exact directives PR #337 just removed. `is-enabled` / `is-active` can't catch this class because the unit would pass them even with the wrong directives in place. Same check for `nss-lookup.target` as a WARN
  - **`check_qos.sh`** ‚Äî `tc qdisc show dev <egress>` for CAKE root qdisc + `iptables -t mangle -L OUTPUT` for DSCP EF on `:1704` + configurable `:$SNAPSERVER_RPC_PORT` + networkd-dispatcher persistence hook at `/etc/networkd-dispatcher/routable.d/50-cake-qos`. Bufferbloat on the LAN is the standard "click between rooms" symptom ‚Äî silent in v0.7.0 if `boot-tune.sh` had failed
  - **`check_system.sh`** ‚Äî cgroup memory controller present (cgroup v1 hierarchy, cgroup v2 unified, OR cmdline opt-in if neither `/sys` layout matched), quiet-boot cmdline flags when `/dev/fb0` is present (`quiet`, `loglevel=3`, `systemd.show_status=false`, `vt.global_cursor_default=0`, `logo.nologo` ‚Äî without these, boot text overlaps fb-display), `/run` tmpfs sized correctly with **RAM-aware floors** (‚â•4 GB Pi: ‚â•1024 MB warn / ‚â•800 MB fail; 1-2 GB: ‚â•200 / ‚â•150; <1 GB: ‚â•80 / ‚â•60 ‚Äî the previous 200 MB hard floor was a false-fail on Pi Zero 2W which PR #221 deliberately sizes ~128 MB) and WiFi/Ethernet exclusivity (`boot-tune.sh` disables WiFi when ethernet has carrier to avoid dual mDNS publication)
  - **`check_timers.sh`** ‚Äî asserts `snapmulti-status.timer`, `snapmulti-diagnostics.timer`, `snapmulti-backup.timer`, `snapclient-discover.timer` are all `enabled` AND `active`. If a timer is missing the gated feature silently degrades: `/status` snapshot is 1 boot old, `/diagnostics` lacks half-hour granularity, MPD db cross-reflash backup never fires, snapclient-discover lost-server failover (PR #285) doesn't run
  - **`check_audio_modules.sh`** ‚Äî for the configured `HAT_CONFIG`, asserts the kernel modules in `lsmod` actually match the chip family (`snd_soc_pcm512x` for PCM5122-based HATs, `snd_soc_wm8804` for S/PDIF HATs, etc.). Without this, a generic codec module + matching card name in `/proc/asound/card0/id` can silently pass the existing Audio section even when the I2S timing or DAC chip is wrong (audio plays at half-speed or with clicks). Includes a `snd_soc_bcm2835_i2s` cross-check for HAT-based audio paths (USB / internal exempt)
- **`scripts/device-smoke.sh` module loader** ‚Äî defines `SCHEMA_VERSION=1`, sources every `scripts/smoke/check_*.sh` it finds at startup, then calls each `check_<name>` function by `declare -F` probe. Modules are mode-aware: those whose first line is `[[ "$MODE" == "server" || "$MODE" == "both" ]] || return 0` skip themselves on client runs, etc. Adding a new check is a single new file in `scripts/smoke/` ‚Äî no edits to `device-smoke.sh` required

Those layers vanish after overlayroot activates. setup.sh now mirrors firstboot's pre-pull guard: if `ENABLE_READONLY=true` and the running driver is not fuse-overlayfs, force the switch (`systemctl stop docker; rm -rf /var/lib/docker/*; systemctl start docker`), wait up to 20 s, verify, and `exit 1` with diagnostic hints if the switch failed. - **HAT detection ran before its own dependencies were installed** ‚Äî `setup.sh` calls `detect_hat` early (line 269) but the apt installs that bring `aplay` (alsa-utils), `i2cdetect` (i2c-tools), and `modprobe` (kmod) happen later via `install_dependencies`. - **snapclient container missing `libasound2-plugins`** ‚Äî display+HAT installs write `defaults.pcm.rate_converter "samplerate_best"` to `/etc/asound.conf` (for `multi_out` ALSA routing to both the HAT and the loopback that feeds audio-visualizer). The plugin lives in `libasound2-plugins`, which the snapclient Dockerfile did NOT install ‚Äî only `libasound2` (the runtime). Without the plugin, ALSA falls back to the default rate converter or fails to open the device. Added `libasound2-plugins` to the runtime image
- **TUI flicker during `--both` install: client install screen briefly overdrew the unified TUI** ‚Äî `progress()` and `progress_init()` in `scripts/common/progress.sh` were missing the `PROGRESS_MANAGED` short-circuit that `milestone()`, `start_progress_animation()`, and `progress_complete()` already had. ### Added
- **`device-smoke.sh`: five new health checks** ‚Äî distilled from the live troubleshooting on 2026-05-07 where each of these conditions had a real-world manifestation in the same hour:

  - **mDNS hostname round-trip** ‚Äî `avahi-resolve -n $(hostname).local` must return one of the host's own IPs (multi-homed Pis with both `eth0` and `wlan0` active are accepted). Catches DHCP/mDNS desync (e.g. IP changed via DHCP renew but avahi cache stale, or hostname mismatch)

  - **Snapcast mDNS strict-client publishing** (server-only) ‚Äî `avahi-browse -rpt _snapcast._tcp` must return at least one `=` (resolved with SRV+TXT) line; PTR-only fails the gate.   - **Hardware throttle / under-voltage** ‚Äî `vcgencmd get_throttled`. Currently-active bits (0x1/0x4/0x8) hard-fail; sticky historical bits (0x10000/0x40000/0x80000) warn loudly. Under-voltage from a marginal PSU/cable is the #1 hardware-side support issue on community Pi installations

- **`device-smoke.sh`: seven Tier-2 health checks** ‚Äî round of follow-ups to the Tier-1 work, each motivated by a class of issues we want surfaced at install time (instead of weeks later as a "things suddenly stopped working" support ticket):
  - **AirPlay mDNS strict-client** (`_raop._tcp`) and **Spotify mDNS strict-client** (`_spotify-connect._tcp`) (server-only) ‚Äî same `avahi-browse -rpt` SRV+TXT verification as the Snapcast check from Tier 1, since shairport-sync and librespot share the same upstream avahi-client lifecycle bug
  - **Avahi socket bind-mount per container** ‚Äî verifies that PR #290 (snapserver) is still in `docker inspect` for the running container, plus mpd and snapclient. A redeploy from an older docker-compose.yml would silently lose the mount and degrade mDNS to PTR-only without this gate
  - **HAT consistency** (client/both) ‚Äî the expected ALSA card is resolved from `SOUNDCARD=hw:CARD=NAME,DEV=0` (headless format introduced by PR #282) OR from `/etc/asound.conf` (`ctl.!default { card NAME }` first, fallback to the first non-Loopback `hw:NAME` reference) when `SOUNDCARD=default` (display+HAT setups using the `multi` ALSA plugin to fan audio to both the HAT and a Loopback for audio-visualizer / fb-display). The resolved name must appear in `aplay -l`. Catches stale or mis-detected HAT configs (e.g. user swapped from HiFiBerry to InnoMaker without re-running setup.sh) and verifies that display+HAT routing didn't lose its DAC binding
  - **Audio path active-params** (client/both, when audio is playing) ‚Äî `/proc/asound/.../hw_params` must show `44100 / S16_LE / 2` when active. INFO when idle (no audio currently flowing ‚Äî not a fail), pass when active and at the snapMULTI invariant format, fail only when active but at the WRONG format
  - **Containerd self-heal counter** ‚Äî `/var/lib/snapmulti-installer/containerd-heal.count` (set by the H5 fix in PR #298). Above 3 ‚Üí fail (structural ENOSPC, not transient). 1-2 ‚Üí warn. 0 / absent ‚Üí pass. Gives operational visibility into the false-ENOSPC family on Pi Zero 2W / Pi 3
  - **Music library non-empty** (server-only, when `MUSIC_SOURCE=nfs|smb`) ‚Äî `find` for known audio file extensions under `MUSIC_PATH`. Local sources (USB, internal disk) deliberately skipped ‚Äî too many legitimate states (empty drive, fresh install, etc.). Network-backed shares are the case where empty = clear configuration error
  - **Snapcast JSON-RPC API** (server-only) ‚Äî `Server.GetStatus` POST to `http://127.0.0.1:1780/jsonrpc` must return 200 with a JSON body containing `"result"`. Smallest functional check that proves snapserver is not just `Up` per docker but actually serving control requests (e.g. for the Android Snapcast app, or our own discover-server.sh)

### Security
- **Tidal Connect: explicit security disclosure** ‚Äî the `tidal-connect` source is built on Raspbian Stretch (EOL 2019-07) with `archive.debian.org` + `trusted=yes` and contains a proprietary, unmaintained binary. The service has always been opt-in via the `tidal` Compose profile, but the trade-off was not documented. `docs/SOURCES.md` now carries an explicit "READ BEFORE ENABLING" block; `README.md` flags Tidal as opt-in with a link to the disclosure; the comment block in `docker-compose.yml` warns at the source. No behaviour change ‚Äî disclosure only. Italian translation synced

### Changed
- **Boot-reliability hardening from 3-council audit (Tier 1)** ‚Äî seven follow-on fixes for the audit-driven launch-readiness work (`CAP-20260506-three-councils-synthesis.md` in vault):
  - `snapmulti-server.service` + `snapclient.service` now declare `RequiresMountsFor=` on their project root and `/audio`, blocking Compose start until the underlying mounts (NFS / USB / fstab) actually attach instead of relying on `_netdev,nofail` + reactive recovery
  - `tune_avahi_daemon` now installs `/etc/systemd/system/avahi-daemon.service.d/network.conf` with `After=network-online.target` + `Wants=network-online.target`, preventing avahi from publishing mDNS on a transient IP that NM revokes seconds later

- **Consolidated apt operations during firstboot** ([#280](https://github.com/lollonet/snapMULTI/pull/280)) ‚Äî Docker apt repo now added before `install-deps.sh`'s `apt-get update`, so a single update covers Debian + Docker sources. `fuse-overlayfs` moved into `install-deps.sh` (gated on `ENABLE_READONLY=true`).

### Fixed
- **`metadata-service.py` — `elapsed` omitted instead of emitted as `0` when service starts mid-track on AirPlay/Tidal**. `_estimate_elapsed()` tracks elapsed via local monotonic clock anchored at "first time we observed the track". For MPD and Spotify that doesn't matter because those sources report their own position natively. For AirPlay and Tidal there is no native position API — if metadata-service is restarted while a track is already playing, the legacy code returned `elapsed: 0` and then accumulated wall-clock time from that point, so the client UI rendered the progress bar at the start of a song that was actually half-way through, with the offset persisting for the rest of the track. Fix: a new `_FRESH_START_GRACE_SEC` window (5 s) distinguishes track-change events caught in real time (calibrated, legacy `elapsed: 0` behaviour) from cold-start mid-track observations (uncalibrated, `elapsed` field omitted from the payload). On the next track change the timer re-calibrates and elapsed resumes. `docs/CLIENT-METADATA.{md,it.md}` "Missing fields" section now documents the contract so client UIs render the absence as "elapsed unknown" (`--:--` or indeterminate progress bar) instead of `0:00`. New `tests/test_metadata_service.py::TestEstimateElapsedCalibration` (7 assertions): grace-window calibration, cold-start absence, post-track-change recovery, per-stream isolation, pause/resume during uncalibrated session.

## [0.8.1] — 2026-06-04

> **v0.8.1 follow-up track** — 7 PRs after v0.8.0 consolidating the SSOT pattern across the installer chain + closing audit follow-ups. Image set unchanged (`0.7.7`), no Docker rebuild; all changes are in the installer chain (`firstboot.sh`, `setup.sh`, `prepare-sd.{sh,ps1}`) + new shared helpers under `scripts/common/`. Validated locally on bash 3.2 (macOS) and bash 5 (CI); fleet reflash gated on this tag. PR series: #586 (overlayroot SIGPIPE), #587 (per-container memory limit on /status), #588 (VERSION bake `--abbrev=0`), #589 (fleet-smoke SSH multiplexing), #590 (container manifest SSOT), #591 (env_get helper SSOT), #592 (drop ensure_overlayroot_initramfs_ready + 4 post-review fixes).

### Changed
- **Overlayroot install flow — install snapmulti-lzma hook BEFORE raspi-config (-292 LOC)**. The pre-PR flow installed the snapmulti-lzma initramfs hook AFTER `raspi-config nonint do_overlayfs 0`, then re-ran `update-initramfs -u -k all` via `ensure_overlayroot_initramfs_ready` so the hook would actually land in `/boot/firmware/initramfs*`. That second rebuild collided with the read-only finalize step remounting `/boot/firmware` ro, producing the cosmetic `cp: cannot create regular file '/boot/firmware/initramfs8': Read-only file system` warnings on every reflash — and the rc-out-of-subshell pipefail bug in PR #586 (`_initramfs_already_has_liblzma`) only existed because of this second pass. Fix: reorder the call sites so `install_initramfs_lzma_hook` runs BEFORE `raspi-config nonint do_overlayfs 0`. raspi-config's own internal `update-initramfs -c -k all` then picks up the hook on its first pass — liblzma lands in initramfs without a second rebuild, the ro-finalize collision becomes moot, and `ensure_overlayroot_initramfs_ready` + the two private helpers it drove (`_initramfs_target_for_kver`, `_initramfs_already_has_liblzma`) were removed from `scripts/common/overlayroot-lifecycle.sh`. Three call sites updated: `scripts/common/system-tune.sh` (server firstboot), `client/common/scripts/setup.sh` (client firstboot), `client/common/scripts/ro-mode.sh` (on-device operator CLI). The previous test `tests/test_initramfs_idempotent_skip.sh` (which pinned the now-deleted idempotency helper) was replaced with `tests/test_overlayroot_lzma_hook_ordering.sh` — 8 assertions: dead helpers actually removed, `install_initramfs_lzma_hook` retained, hook-call-line precedes raspi-config-call-line in all three callers, no uncommented residual `ensure_overlayroot_initramfs_ready` calls anywhere. Net change: -292 LOC (-433/+141), one fewer initramfs rebuild round per install, and the cosmetic ro-finalize warning that confused every operator is gone at the root.

### Added
- **`scripts/common/env-reader.sh` — SSOT helper `env_get` for runtime `.env`-style reads**. The v0.8 hardening track introduced `install_conf_get` for `/boot/.../install.conf`, but runtime `.env` parsing was still duplicated across 5 modules with subtle variants of `grep -m1 '^KEY=' "$FILE" | cut -d= -f2- | tr -d ...`. Variants drifted on quote-handling, CR/whitespace stripping, and values containing `=` (passwords, base64, query strings). New helper `env_get KEY FILE [STRIP_MODE]` is Bash 3.2 compatible, validates KEY as a POSIX identifier, returns empty + rc 0 on missing file/key, and supports 4 explicit strip modes (`none` / `cr` / `trim` / `all`) so the caller declares its intent at every site. Always uses `cut -d= -f2-` so values containing `=` survive. Migrated five low-risk consumers — `scripts/common/play-smoke-tone.sh` (SNAPMULTI_BOOT_SMOKE_TONES, strip=all), `scripts/smoke/check_system.sh` (SNAPMULTI_RELEASE / SNAPMULTI_IMAGE_SET, strip=none), `scripts/smoke/check_mounts.sh` (MUSIC_SOURCE / MUSIC_PATH, strip=trim), `scripts/smoke/check_qos.sh` (SNAPSERVER_RPC_PORT, strip=trim), `scripts/diagnostic.sh` (release identity, strip=none). All five sites guard the call with `if declare -F env_get` so stripped / legacy bundles fall back to their original inline form (zero behaviour regression). Installer paths (`deploy.sh`, `setup.sh`, `firstboot.sh`) intentionally NOT migrated in this PR — higher-risk surfaces deferred until env_get has runtime miles. New `tests/test_env_reader.sh` (49 assertions): contract (input validation, missing-file/key tolerance, value-with-`=` preservation, duplicate-key first-wins), all 4 strip modes against fixture rows with quotes / CRLF / trailing spaces / tabs, unknown-mode rejection, plus migration pinning (every consumer wires `env_get` AND the guarded fallback). Staged through all 6 SD-prep sites per v0.8 6-site rule; `tests/test_prepare_sd_required_files.sh` 39 → 44 → 50 to pin staging end-to-end. Does NOT replace `install_conf_get` — install-conf parsing stays separate from runtime `.env`.
- **`scripts/common/container-manifest.txt` — SSOT for snapMULTI container names + roles**. PR #587 introduced two parallel hardcoded lists: `_SNAPMULTI_CONTAINERS` in `scripts/smoke/check_containers.sh` and `_CONTAINER_ROLE` in `docker/metadata-service/metadata-service.py`. Adding a container to docker-compose.yml then required editing both lists, in two languages, and any drift would land silently — the smoke would silently skip the new container, or `/status` would group it under the wrong header. The manifest is a whitespace-separated `<name> <role>` flat-text file (no jq / no Python dependency on the bash side) consumed by a new `_load_container_manifest` loader in the smoke module and a new `_load_container_role_manifest` loader in the Python module. Both keep their pre-manifest hardcoded fallback so legacy / stripped bundles still render; new `tests/test_container_manifest.sh` (11 assertions) enforces that the manifest is well-formed, that both loaders are actually called (no silent fallback regression), that the loaded shape matches the manifest exactly, and that no bare-literal duplicate of the full list survives in either consumer. Staged through all 6 SD-prep sites (`STAGING_COMMON_SHARED_MODULES` + `prepare-sd.sh` top-level + client verify + `prepare-sd.ps1` `$requiredBase` + client requiredFiles + `$shared` selective-copy foreach) per the v0.8 6-site staging rule; `tests/test_prepare_sd_required_files.sh` extended (39 → 45 assertions) to pin the staging. `docker-compose.yml` bind-mounts the file into the metadata container at `/app/container-manifest.txt:ro` so the Python loader finds it without an image rebuild. No behaviour change on existing fleet: bash + Python sides now consume the same SSOT, both produce the same name → role set they did before.

### Fixed
- **`scripts/fleet-smoke.sh` collapsed every host into `UNREACH ssh-timeout-or-fail`** when the operator's local SSH config used multiplexing (`ControlMaster auto` + `ControlPath ~/.ssh/sockets/...`). On sandboxed / dev shells the socket-path bind fails, the script's `2>/dev/null` swallowed the actual stderr (`unix_listener: cannot bind to path`), and every distinct failure — host-key-changed, auth-failed, connection-failed — degraded to the same opaque blob, forcing the operator to chase ghosts. Fix: (1) bake `-o ControlMaster=no -o ControlPath=none` into `SSH_OPTS` so the probe is independent of the operator's global SSH state (host-key validation stays at `accept-new` — NO `StrictHostKeyChecking=no`); (2) capture SSH stderr per host into a tmp file and classify it via two new helpers — `_classify_ssh_stderr` returns one of `host-key-changed`, `ssh-controlpath-error`, `auth-failed`, `connection-failed`, `smoke-timeout`, `ssh-failed`, and `_ssh_failure_note` emits an actionable fix when one exists (e.g. host-key-changed → `run: ssh-keygen -R <host> && ssh-keygen -R <ip>`). The text table's NOTES column dropped its 30-char truncation so the suggested command is verbatim copy-pasteable. New `tests/test_fleet_smoke_ssh_classification.sh` (18 assertions): SSH_OPTS multiplexing-bypass + host-key validation preservation pinned statically, classifier exercised against real-shape stderr samples, note builder verified for IP / no-IP host-key cases. Bash 3.2 + bash 5 + shellcheck clean.
- **`prepare-sd.{sh,ps1}` VERSION bake masked post-tag commits** — both installers staged `server/.version` + `client/VERSION` using `git describe --tags --abbrev=0`. `--abbrev=0` strips the `-<N>-g<sha>` distance suffix, so a flash from a main HEAD that was N commits past the latest tag baked the BARE tag onto the SD card. Result: `/status`, fb-display footer, and the diagnostic bundle all reported the device as the bare release (e.g. `v0.8.0`) even though the running code carried N additional commits. Fix: drop `--abbrev=0` from both installers — `git describe --tags` automatically appends `-<N>-g<sha>` when HEAD is not on a tag (e.g. `v0.8.0-2-g7f5601d`), and returns the bare tag when it is. Cross-platform parity check pinned in `tests/test_prepare_sd_version_bake.sh` (7 assertions: invocation present, no residual `--abbrev=0`, rationale comment within 12 lines on bash, same on ps1).

### Changed
- **`/status` Containers section: per-row memory limit (client + server)** — `check_containers.sh` now reads `HostConfig.Memory` from the running container (already done for the drift-detection check) and appends `(limit=64M)` to the smoke message. `metadata-service.py`'s `_CONTAINER_PATTERN` parses the optional suffix into the `desc` column so each row on the /status page surfaces the actually enforced limit next to the health badge. Runtime-driven — the source of truth is Docker engine state via `docker inspect`, NOT propagated env vars. Net effect on a both-mode host: snapclient / audio-visualizer / fb-display rows now show their limits alongside snapserver / mpd / mympd / metadata etc., without any cross-`/opt` `.env` propagation, Docker socket bind-mount, or new env var pass-through in `docker-compose.yml`. Backward-compat: snapshots without the suffix still render exactly as before (regex `(limit=…)?` is optional; classification + `desc=""` unchanged for legacy rows). Conservative: missing `numfmt`, `HostConfig.Memory==0`, or a non-integer value all leave the suffix off — the row still renders, only the limit column is omitted.

### Fixed
- **overlayroot: `_initramfs_already_has_liblzma` SIGPIPE-pipefail false-negative** — the helper used `lsinitramfs "$target" | grep -qF "liblzma.so.5"`; with `set -euo pipefail`, `grep -q` exits at the first match, the upstream `lsinitramfs` (streaming ~10k entries from a ~12 MB cpio archive) receives SIGPIPE and exits 141, pipefail propagates the 141 to the helper, and the check always returned false. Result: `ensure_overlayroot_initramfs_ready` always took the "refreshing" branch on every kernel, the second-and-later kernels ran `update-initramfs` after `/boot/firmware` had already been remounted ro by the read-only setup step, and firstboot logged the cosmetic `update-initramfs -u -k 6.18.33+rpt-rpi-… failed` + `first boot may not activate overlay` WARN trio on every fresh install. Overlay still came up because the dpkg trigger had already produced a usable initramfs into `/boot/firmware` (kmod on trixie pulls liblzma via its linker dependency), so the WARN was load-bearing only as noise — but it did make `/var/log/snapmulti-install.log` look broken. Fix: switch to `grep -F "liblzma.so.5" >/dev/null` so the consumer drains the entire stream and `lsinitramfs` exits 0. Regression test added in `tests/test_initramfs_idempotent_skip.sh` (fake `lsinitramfs` producing 10k+ lines around the match, asserting helper succeeds under `set -euo pipefail`).

All three now validate the remount via `findmnt` and exit 0 with a journal entry when the partition is still ro — best-effort backup, transient failure does not degrade the overall system health signal. Root cause of the original remount failure remains under investigation; this fix decouples backup transient errors from `/status` fail signaling.
- **metadata-service `/version`: semver tuple comparison for `update_available` instead of `!=`** — observed on snapvideo with `SNAPMULTI_RELEASE=v0.7.9.5` in `.env` while GitHub `releases/latest` returned `v0.7.9.3` (v0.7.9.4 + v0.7.9.5 were tag-without-publish). The old `latest != current` flagged `update_available=true`, surfacing the misleading "Update available: 0.7.9.5 -> 0.7.9.3 (reflash to apply)" on the `/status` page. `_parse_version_tuple()` now converts both sides to int tuples and the check becomes `latest_tuple > current_tuple`. Pre-release tags / dev clones (non-numeric parts) fall back to the old `!=` comparison so the endpoint never crashes. 11 new regression tests pin the matrix (current ahead, current behind, equal, unparseable, major/minor jumps).
- **install-deps: `apt-get upgrade` → `apt-get dist-upgrade` so kernel + rpi-eeprom are not kept back** — plain `upgrade` refuses to install new packages or remove old ones, so meta-packages like `linux-image-rpi-2712` / `linux-image-rpi-v8` / `rpi-eeprom` are silently held back when the new version requires installing a new sub-package (the new kernel image deb). Observed on snapvideo 2026-05-29: 6.12.75 → 6.18.29 kernel + 28.15 → 28.24 eeprom all stuck after a clean reflash. Fix applies to both `install-deps.sh` and the final upgrade pass in `firstboot.sh`. `apt-get dist-upgrade -y` is the standard for "actually apply pending kernel + eeprom updates".
- **install-deps: drop `dstat` from monitoring tools (eliminates pcp entirely)** — `dstat` Depends on `python3-pcp`, which transitively pulls Performance Co-Pilot (~50 MB, multi-daemon). On Debian trixie neither `sysstat` nor `iotop-c` Depends on pcp (verified: `apt-cache depends iotop-c` shows only libc6, libncursesw6, libtinfo6), so dropping `dstat` means new installs no longer pull pcp at all — no transient install, no `systemd-sysv-generator: SysV service '/etc/init.d/pcp'` journal noise. `sar` from sysstat covers dstat's realtime CPU/mem/disk use case.
- **install-deps: explicit `apt-get purge pcp pcp-conf 'pcp-pmda-*'` removed (dead code after dropping dstat)** — with pcp no longer pulled by any package we install, the trailing purge has nothing to clean. The existing `apt-get autoremove --purge -y` (run earlier in the same function) acts as the safety net for any orphan pcp left over from a base Pi OS image or a stale state. -7 LOC, comment block rewritten to document the new invariant.

## [0.7.9.1] — 2026-05-28

> Script-only patch (image_set stays 0.7.7). Resolves the v0.7.9 persistence regression. Field-validated on snapvideo: overlay active + `server.json` + `server.json.prev` + 4 groups all survived reboot under `overlayroot="tmpfs"`.
>
> **Known limitation:** very large NFS/SMB libraries (~76 k+ tracks observed) can still exceed the install healthcheck window even with the new 120 s stability grace introduced in this release. When that happens, `firstboot.sh` writes `.install-failed`, the `[finalize]` step never runs, and the device boots on plain ext4 (no overlay protection) — even though containers are healthy and the LAN-side appears fine. Pre-set `MPD_START_PERIOD=3600s` in `install.conf` BEFORE flashing the SD card. See [docs/ADVANCED.md — MPD_START_PERIOD](docs/ADVANCED.md#mpd_start_period) and [docs/TROUBLESHOOTING.md — Install marked failed but containers run](docs/TROUBLESHOOTING.md#install-marked-failed-but-containers-run).

### Fixed
- **v0.7.9 persistence regression cascade (#527)** — multiple compounding bugs from the v0.7.9 persistence work, all corrected here. The original `snapmulti-data-persistence.service` bind-mounted `/opt/snapmulti/data` over `/media/root-rw/snapmulti-persist/`, but `/media/root-rw/` is itself tmpfs on `overlayroot="tmpfs"` — the bind worked at runtime but the persistent location was wiped at reboot, and the smoke check `[OK] Persistent bind active` actively masked the failure. Replaced with the proven `/boot/firmware/`-backup pattern (FAT32, the only userland-writable path that survives reboot on tmpfs overlayroot):
  - **Backup trigger** is split: `snapmulti-state-backup.path` watches myMPD workdir (event-driven, ~1 s latency for playlist/theme/script edits); `snapmulti-state-backup.timer` runs every 5 min for `server.json` (snapserver rewrites that file every ~3 s to refresh client `lastSeen` heartbeats — wiring it to `.path` would cause `~354` backup-service invocations per 5 min and churn the FAT32 boot partition). `backup-snapmulti-state.sh` short-circuits on canonical equality (sort keys + strip `lastSeen` + sha256 compare) so a no-op timer tick costs only one JSON parse — zero FAT32 writes when only heartbeats changed.
  - **Restore** runs as `ExecStartPre` on `snapmulti-server.service` BEFORE compose up, so snapserver finds restored state instead of writing a fresh empty one. The script PUID/PGID is read from `/opt/snapmulti/.env` for non-default user installs.
  - **Scope** matches the original bind-mount: snapserver `server.json` + WHOLE `myMPD/workdir/` (state + config + smartpls + scripts + pics). Narrowing to `state/` alone would silently lose user-customised smart playlists / theme / scripts on reboot.
  - **Seed on install** runs `snapmulti-state-backup.service` once when arming the watcher — protects upgrade scenarios where state pre-exists the watcher and would otherwise leave the boot-partition backup stale until the next user change.
  - **Boot partition mount state preserved** — `backup-snapmulti-state.sh` + `backup-mpd.sh` detect whether `/boot/firmware` was previously ro and restore THAT state on exit. The original v0.7.9 fix unconditionally remounted ro at exit, which during firstboot (both-mode) cascaded to break `raspi-config do_overlayfs` and prevented overlay activation entirely — discovered by user pattern-recognition ("readonly in server mode has always worked").
  - Removed `snapmulti-data-persistence.service`, `snapmulti-data-setup.sh`, and the misleading `check_persistence.sh` smoke module; replaced with `check_state_backup.sh` (verifies backup freshness on the actual persistent path) + `check_timers.sh` now also asserts the path + timer units are enabled and active.
  - **`backup-from-sd.sh` catches up to widened backup scope (#530)** — host-side reflash extractor now handles the current `mympd/workdir/` layout (atomic stage+swap to avoid merging stale files from the previous SD with the restored backup) and restores `data/server.json`. Legacy `mympd/state/`-only backups still work with a warning that user customisations aren't in the backup.
  - **Backup integrity + `.prev` rotation** — `backup-snapmulti-state.sh` validates each `server.json` write (size ≥ 64 B + JSON parse via `jq` when available) before publishing, and rotates the previous good copy to `server.json.prev` so a single transient corruption (truncated write, FAT32 fragmentation, power loss mid-publish) cannot brick the device. `restore-snapmulti-state.sh` validates `server.json`; falls back to `.prev` with a WARN log if current is corrupt; fails loud only when BOTH are unparseable. Plus a post-`verify_services` seed in `deploy.sh` so the boot-partition backup captures real state on first install (the in-place seed at install time was a no-op on fresh installs because snapserver hadn't written yet). 2 new bash tests (`test_backup_integrity_validation.sh`, `test_restore_prev_fallback.sh`) lock the lessons in CI.
- **`verify_services` healthcheck stability grace (120 s)** — Docker does not necessarily mark a container healthy the instant the inner service binds its port; the next health probe runs up to 30 s later, and `verify_compose_stack` also needs a second sample to prove `RestartCount` stability. Large NFS libraries were observed to bind just after the 300 s start_period and then fail `verify_services` seconds before the next successful probe. New explicit `health_grace_seconds=120` on top of `max(MPD_START_PERIOD, 180s)` covers that window. The install log now exposes the full budget (`up to ${verify_budget}s, ${wait_seconds}s interval; MPD start_period=${start_period}s + 120s grace`). Does not eliminate the need to bump `MPD_START_PERIOD` for very large libraries — see Known limitation above.
- **Mount smoke accepts active `.mount` as valid (`check_mounts.sh`)** — some systemd versions mark the `.automount` companion inactive once the actual `.mount` has fired (e.g. after MPD scanned the library). Previously this scored as a fail. Now passes when either the automount is watching OR the mount has already fired; only fails when neither holds.

>
> **Known issue (fixed in [0.7.9.1]):** `snapmulti-data-persistence.service` shipped in this version does NOT actually persist on the standard `overlayroot="tmpfs"` setup. The smoke check `[OK] Persistent bind active` is misleading. See #527 + the [0.7.9.1] entry above for the proper fix.

### Added (FAILED — see Known issue)
- ~~**snapserver + myMPD state persistence (`snapmulti-data-persistence.service`, #525)**~~ — architecturally broken on tmpfs-mode overlayroot. The bind-mount source `/media/root-rw/snapmulti-persist/` was itself tmpfs, wiped at reboot. Replaced in [0.7.9.1] with `/boot/firmware/` backup pattern.
- **Status page coverage expanded (#525)** — 6 new checks surface previously-invisible state:
  - `/boot/firmware` FAT32 free space (fixed 512 MB partition — full = no diagnostic bundles, no MPD backup, no kernel upgrade)
  - `/media/root-rw` backing fs free space (full = overlay cannot grow, OOM imminent)
  - Memory headroom + `MemAvailable` (would have caught the mympd OOM in v0.7.8.16 hours earlier — surfaces 80 %+ pressure)
  - Swap pressure (heavy swapping degrades audio path latency)
  - Pending update (queries `/version` endpoint — operator sees at a glance if a reflash brings a newer release)
  - Now-playing stream + track (which stream the connected clients actually hear, plus track metadata)
  - Previous boot end state (info-level — flags kernel panic / power loss vs clean reboot)

### Changed
- **IPv6 disabled by default on every install (ADR-007, #522)** — `prepare-sd.sh` / `prepare-sd.ps1` now write `ipv6.disable=1` to `/boot/firmware/cmdline.txt` by default. snapMULTI is a LAN-only audio appliance and dual-stack mDNS / Snapcast discovery causes intermittent silent failures on consumer LANs (see #521 + the broader Avahi dual-publish race in #425). Kernel-level disable pre-empts every userland subsystem, lives outside the overlayroot upper layer, and is reversible by removing the token from `cmdline.txt`. Opt out with `DISABLE_IPV6=false ./scripts/prepare-sd.sh ...` for the rare operator with a hard IPv6 requirement. Documented in [ADVANCED](docs/ADVANCED.md#ipv6-disabled-by-default) + [TROUBLESHOOTING](docs/TROUBLESHOOTING.md#ipv6-disabled--is-this-a-problem). `install-deps.sh` installs `Acquire::ForceIPv4 "true"` apt config so first-boot package fetches don't stall on AAAA timeouts.

### Fixed
- **Pi Zero 2W — snapclient IPv6 link-local latch (#521 → #522)** — snapclient's built-in libavahi-client browse occasionally latched onto an IPv6 link-local SRV target that did not route on the local LAN, leaving audio silent with a healthy server visible to every other client. #521 shipped a surgical workaround (`ExecStartPre` pin `--host <IPv4>` into `/etc/default/snapclient`). #522 supersedes it by killing IPv6 at the kernel level — the workaround is removed, snapclient's built-in mDNS auto-rediscovery is restored for free, and the same class of bug stops affecting other consumers (fb-display, future zeroconf code).
- **mympd crash-loop after ADR-007** — `ghcr.io/jcorporation/mympd` upstream default is `MYMPD_HTTP_HOST=[::]:8180` (dual-stack listen). With IPv6 disabled at kernel level the bind returns `EAFNOSUPPORT` (errno 97) and the container exits, re-spawning every few seconds. Pin `MYMPD_HTTP_HOST=0.0.0.0` in `docker-compose.yml`. Caught immediately after rolling ADR-007 to snapvideo. snapserver / MPD / metadata-service already bind IPv4 explicitly — mympd was the only consumer of the upstream IPv6 default.

### Removed
- **PR #521 workaround retired (#522)** — `discover-server-native.sh` + the `ExecStartPre=+/usr/local/sbin/snapmulti-discover-snapserver` line on the Pi Zero 2W drop-in. Native auto-rediscovery restored.
